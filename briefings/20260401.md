## 工程工具链

### 一、Anthropic 在北京时间 4 月 1 日发布 Claude Code v2.1.89，重点补齐 headless 权限治理、MCP 启动等待与长会话稳定性

1. `anthropics/claude-code` 官方 GitHub Release `v2.1.89` 的 `published_at` 为 `2026-04-01T01:07:06Z`，折算北京时间为 `2026-04-01 09:07:06`，时间边界成立。
2. 这一版最关键的新增落在 agent 治理链路：`PreToolUse` hooks 新增 `"defer"` 权限决策，headless 会话可以在工具调用处暂停，再用 `-p --resume` 继续；同时新增 `PermissionDenied` hook，允许外部治理层在自动模式拒绝后返回 `{retry: true}` 触发重试；`MCP_CONNECTION_NONBLOCKING=true` 则允许 `-p` 模式跳过 MCP 连接等待，并把 `--mcp-config` 服务连接上限收敛到 5 秒。
3. 修复项集中指向生产可用性：`Edit(//path/**)` 与 `Read(//path/**)` 现在检查解析后的 symlink 目标而不是请求路径；Windows 下 `Edit/Write` 不再把 CRLF 翻倍或吞掉 Markdown 硬换行；`StructuredOutput` schema cache bug 曾导致多 schema 场景约 50% 失败率，本版修复；同时还修复了 autocompact thrash loop、prompt cache miss、`--resume`/deferred tool 恢复挂起、LSP crash 后 zombie 状态等长会话问题。

2. 为什么值得工程团队关注：这不是普通 UI 微调，而是在把 Claude Code 从“交互式可用”继续推进到“可接企业治理链路”。`defer`、`PermissionDenied`、非阻塞 MCP 连接，分别对应外部审批、失败后重试、以及批处理/CI 启动时延控制；而 symlink、CRLF、schema cache、autocompact 等修复，则直接影响真实仓库里的文件安全、跨平台一致性和长会话稳定性。

3. 最可能影响的环节：开发 / 测试 / 流程 / 安全治理。

【工程意义】
- 对开发：headless 或半自动流程终于可以在工具调用点暂停并等待外部审批，不必只靠“全放行”或“全阻断”两种粗粒度策略；非阻塞 MCP 连接也更适合批处理和脚本化调用。
- 对测试：应补回归覆盖 symlink allow rule、Windows CRLF 写回、多 schema structured output、长会话 autocompact、`--resume` 恢复和 MCP 连接超时，避免“少数边界条件直接拖垮整轮任务”。
- 对流程：平台/安全团队可以开始把 `PreToolUse` / `PermissionDenied` hook 接进内部审批、审计或策略引擎，而不是把 agent 权限控制留给终端使用者手工判断。

【建议动作】
- 立即可做：梳理你们当前编码 agent 的工具审批链，确认是否需要在 headless 流程中引入“暂停-外部审批-恢复”的控制点，而不是仅依赖静态 allowlist。
- 本周可验证：用真实仓库做一组 focused regression，覆盖 symlink 路径规则、Windows CRLF、MCP 启动超时、长会话 compaction、`--resume` 恢复和多 schema structured output。
- 暂时观察：继续观察 `v2.1.90+` 是否沿同一方向继续收敛；若团队大量依赖 headless/CI 或 MCP 工具链，可优先灰度到高频使用者。

【来源】
- GitHub Releases：anthropics/claude-code `v2.1.89`

### 二、Microsoft 在北京时间 4 月 1 日发布 Copilot Chat v0.42.2，修 fetch 失败时的错误编辑产出，并保住 external ingest 场景下的 code search

1. `microsoft/vscode-copilot-chat` 官方 GitHub Release `v0.42.2` 的 `published_at` 为 `2026-03-31T22:27:25Z`，折算北京时间为 `2026-04-01 06:27:25`，时间边界成立。
2. 官方 release note 给出的两条实质变更都偏工程底座：其一，**don't yield edits when fetch errors or is cancelled**，也就是当 fetch 失败或用户取消时，不再继续产出后续编辑结果；其二，**still enable code search when only external ingest is available**，即只接入 external ingest 的环境里仍保持 code search 可用。
3. 这说明 Copilot Chat 0.42.2 的目标不是增加新入口，而是收紧两个容易被忽略的真实问题：检索/拉取链路失败后，agent 不应继续基于不完整上下文给出编辑；而企业索引架构如果走 external ingest，也不该把 IDE 内代码搜索能力一并打残。

2. 为什么值得工程团队关注：这类补丁表面不大，但直接关系到 IDE agent 是否会在“上下文已失真”的情况下继续修改代码，以及企业接入自建/外部索引时是否还能保留搜索能力。对已经把 Copilot Chat 嵌入日常开发的团队，这比多一个实验按钮更贴近真实风险控制。

3. 最可能影响的环节：开发 / 测试 / 流程。

【工程意义】
- 对开发：如果 fetch 被取消或失败，IDE 不再继续吐出潜在错误编辑，能减少“明知上下文缺失却还生成改动”的隐性事故。
- 对测试：应重点补回归覆盖网络中断、人工 cancel、部分检索失败、仅 external ingest 可用时的 code search 可达性，以及这些场景下编辑建议是否被正确抑制。
- 对流程：采用 external ingest 或自建索引通道的团队，可以把“检索失败时编辑行为”和“索引架构切换后搜索能力是否保留”列为验收项，而不是只看模型回答质量。

【建议动作】
- 立即可做：盘点当前 Copilot Chat 或同类 IDE agent 在 fetch 失败/取消时的默认行为，确认是否存在“检索失败仍继续建议改动”的风险。
- 本周可验证：构造一组网络抖动、请求取消、external ingest-only 的场景回归，检查 code search 是否仍可用，以及编辑流是否在失败后及时中止。
- 暂时观察：这是维护性小版本，适合先在使用 external ingest 的团队或高频 IDE agent 使用者中灰度验证，再决定是否广泛推广。

【来源】
- GitHub Releases：microsoft/vscode-copilot-chat `v0.42.2`

### 三、Orca 在北京时间 4 月 1 日发布 v1.0.70，补 hidden files quick-open、更新前终端缓冲保留和 updater 版本比较加固

1. `stablyai/orca` 官方 GitHub Release `v1.0.70` 的 `published_at` 为 `2026-03-31T23:47:09Z`，折算北京时间为 `2026-04-01 07:47:09`，时间边界成立。
2. 本版新增 `allow search for hidden files in quick-open`，意味着 quick-open 开始支持检索隐藏文件；同版还修复了 `capture terminal buffers before quitAndInstall`，在自更新前先抓取终端缓冲；以及 `harden updater version comparisons`，加固更新器版本比较逻辑。
3. 这三个点都很“工程日常”：AI 工作台里最常需要找的文件之一恰恰是隐藏配置，如 `.env*`、`.github/*`、`.claude/*`、`.cursor/*`、`.devcontainer/*`；而自更新前保留终端输出、修正版本比较，则直接影响问题复盘和升级稳定性。

2. 为什么值得工程团队关注：多终端、多仓库 AI 工作台要真进团队流程，关键不只是模型接得多，而是能不能稳当地碰到真实工程文件和真实升级链路。hidden files quick-open 解决的是“配置文件找不到”，终端缓冲保留和版本比较加固解决的是“更新后问题没法复盘”和“版本判断出错”。

3. 最可能影响的环节：开发 / 测试 / 流程 / 选型。

【工程意义】
- 对开发：查找隐藏配置文件会更直接，尤其适合频繁切 `.github/workflows`、`.env.example`、agent/rules 配置和本地开发容器文件的团队。
- 对测试：应补跨平台 quick-open 回归，覆盖隐藏文件、点目录、工作区根/子目录差异，以及 quit-and-install 前终端缓冲保留和版本比较边界。
- 对流程：如果团队在评估多终端 AI 工作台，应该把“能否可靠操作隐藏工程配置”和“升级链路是否可回溯”视为基础验收项，而不是只看聊天和补全能力。

【建议动作】
- 立即可做：把 `.github`、`.env*`、`.claude`、`.cursor`、`.devcontainer` 等路径加入 quick-open 与文件导航回归样本，不要只测普通源码目录。
- 本周可验证：做一次真实的 quit-and-install 演练，确认更新前能否保留终端日志、更新后版本识别是否正确，以及隐藏文件搜索是否覆盖团队常用配置路径。
- 暂时观察：继续观察 1.0.7x 后续版本是否仍围绕文件导航、更新器和焦点/终端稳定性补丁收敛；如果是，说明产品正在往团队可用性而不是单点功能堆叠推进。

【来源】
- GitHub Releases：stablyai/orca `v1.0.70`
