## 工程工具链

### 一、OpenAI Codex CLI 0.125.0 GA：Permission Profile 全链路打通、`codex exec --json` 报告 reasoning token、远程 plugin marketplace 与 Unix socket 传输上线

1. OpenAI 4 月 24 日 18:00 UTC（北京时间 4 月 25 日 02:00）将 Codex CLI 从 0.124.0 跳到 `rust-v0.125.0` 正式版，对应 npm `codex@0.125.0` 与全平台二进制；同期 0.126.0-alpha.1 / alpha.2 在 BJT 4-25 凌晨与上午发出，但仍是 prerelease，团队正式上线请固定到 0.125.0。
2. App-server 层一次性补齐四件事：① 新增 Unix socket transport（替代仅支持 TCP 的旧链路）；② thread/resume、thread/fork 增加 `excludeTurns` 与分页友好的恢复语义，长会话续传不再会被一次性拉爆；③ 新增 sticky environment / 多环境切换 API，允许同一个 session 在多 workspace、多远程沙箱之间稳态绑定；④ 新增远程 plugin marketplace 的 install / upgrade RPC，企业内网私有 marketplace 可走标准接口。
3. Permission Profile 现在贯穿 TUI session、user turn、MCP sandbox state、shell 升级与 app-server API 全链路 round-trip——同一份"沙箱+权限"声明可以在 IDE、CLI、远程 session 间一致传递，不再因为换路径就被回退到默认；模型方提供商正式接管模型发现，Codex 客户端可以从 `account/read` 拿到 AWS / Bedrock 账户状态；`codex exec --json` 现在会在 usage 字段里输出 reasoning token 数，便于编程方按真实推理量计费与监控；rollout tracing 现在记录 tool / code-mode / session / multi-agent 边界，并新增 debug reducer 命令做轨迹回放。

为什么值得工程团队关注：这一组变更不是表层 UI 调整，而是把 Codex 从"单机 CLI"扩成"可远程托管的 agent runtime"，对所有把 Codex 接入内部平台、CI、Slack/Teams bot 或自建 agent 网关的团队是一次稳定接口窗口。Permission Profile round-trip 解决的是"agent 在多个执行面之间权限一致性"这一长期痛点，对安全与合规审计是直接利好；reasoning token 字段则是预算治理终于能拿到的客观度量。

最可能影响的环节：开发 / 流程 / 平台。

【工程意义】
- 对开发：改用 Unix socket transport 可避免本机端口冲突与 TCP 监听被防火墙拦；多环境 sticky API 让 IDE 插件 / 自研壳层不必每个 turn 重传 cwd 与 env；`codex exec --json` 的 reasoning token 字段可直接接进现有 token 监控。
- 对测试：Permission Profile 全链路一致后，可以为"高风险任务"统一在 profile 文件里收紧 sandbox / shell 升级权限，不必担心远程 session 自行放宽；rollout trace 的 debug reducer 让回放复现 agent 失败序列变得可行，长链路 agent 回归测试能落到自动化。
- 对流程：远程 plugin marketplace install/upgrade RPC 让"内部插件目录"具备可治理的下发机制，配合 managed settings 的 `blockedMarketplaces` / `strictKnownMarketplaces`（同期 Codex 0.124 已做强制）可形成合规闭环。

【建议动作】
- 立即可做：将团队内的 Codex 安装版本固定到 `0.125.0` 而不是 `0.126.0-alpha.*`，并在 CI / 镜像构建里显式 pin；测试 `codex exec --json` 输出的 usage.reasoning_tokens 字段是否已能被现有监控管线消费。
- 本周可验证：把内部沙箱策略整理成一份 Permission Profile，验证它在 TUI、`codex exec`、MCP sandbox、远程 app-server 四个面是否表现一致；用 Unix socket transport 替换一处现有 TCP app-server，并观察 WebSocket 通知拥塞是否缓解（本版本同步增大了 outbound buffer）。
- 暂时观察：0.126.0 alpha 引入的新 sticky env / remote thread store 接口仍在打磨，等待 0.126.0 GA 后再迁移内部 agent 编排。

## 生成式AI

### 二、Anthropic 上线 Rate Limits API：管理员可编程查询组织与每个 workspace 的 RPM / TPM / Spend 限额

1. Anthropic Claude API release notes 在 "April 24, 2026" 这条新增 Rate Limits API：管理员可通过编程方式查询组织与每个 workspace 当前生效的 rate limits 与 spend limits（按时区默认折算 PT，对应北京时间 4 月 25 日凌晨）；首批端点放在 `docs.anthropic.com/docs/en/build-with-claude/rate-limits-api`，需要 Admin API 凭据。
2. 这里查询的是"服务实际生效的当前限额"，而非历史用量——历史用量与花费仍走 2025 年 8 月上线的 Usage & Cost API。两套 API 拼起来才能完整回答"这个 workspace 现在还能跑多少 tokens、最近 7 天烧了多少钱、距离上限还有多远"。
3. Anthropic 此前一直只允许在 Console 的 Usage 页面里看每模型 tier 的 RPM / 输入 token / 输出 token 与 spend 阈值，工程上要做"workspace 级配额护栏"必须人肉抄；本次新增让多团队、多业务共用一个组织、按 workspace 分摊预算的场景，第一次具备了 API 化治理的基础。

为什么值得工程团队关注：这是 Anthropic 用户长期反馈最高的工程缺口之一——尤其是把 Claude 接进内部 agent 平台、按业务/团队拆 workspace 收费的公司。今日上线后，"workspace 配额监控 + 触发预警"可以做成 Prometheus / Datadog 指标而不是人工巡检；同时也是构建多租户网关时做"剩余配额回退到备模型"逻辑的前提条件。

最可能影响的环节：流程 / 平台 / 成本治理。

【工程意义】
- 对开发：内部 LLM 网关可在请求前先查一次 workspace 当前 RPM / TPM 上限，做客户端侧令牌桶而不是只靠 429 后退。
- 对测试：把 Rate Limits API 的查询纳入"每日预算监控"健康检查，防止 workspace 上限被静默调整后无人知晓。
- 对流程：把"workspace 限额变更"做成审计事件，结合 Admin API 的成员/凭据查询，把多业务共用 Claude 的成本边界制度化。

【建议动作】
- 立即可做：用 Admin API 凭据调一次 Rate Limits API，校验现有 workspace 的 RPM / 输入 TPM / 输出 TPM 与 monthly spend 上限是否与 Console 记录一致；不一致即排查最近的人工修改记录。
- 本周可验证：把 Rate Limits API 的关键字段接进现有 LLM 监控盘（建议至少包含 RPM、input_tpm、output_tpm、monthly_spend_limit、剩余预算比例），并在到达 80% / 95% 时触发告警。
- 暂时观察：本次只放了"查询"端点，"修改 workspace 限额"的 Admin API 仍未公开（社区 #371 仍在 open），不要现在就把"自动调整 workspace 限额"做进自动化预案。

【来源】
- Anthropic Claude Platform Release Notes：`April 24, 2026 — We've released the Rate Limits API`
- Anthropic Docs：`docs.anthropic.com/docs/en/build-with-claude/rate-limits-api`
