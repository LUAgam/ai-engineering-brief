## 工程工具链

### 一、Cursor 上线自托管 cloud agents，把代码、制品和工具执行留在企业内网

1. Cursor 官方 Changelog 页面明确标注为 `Mar 25, 2026`，并同步发布官方博客说明，时间边界成立。
2. Cursor 宣布支持 self-hosted cloud agents：agent 仍由 Cursor 负责编排，但代码库、构建产物、密钥和工具调用可以完全留在企业自有基础设施内执行。
3. 官方说明强调自托管形态仍保留隔离环境、多模型 agent harness、插件等能力，并提供 `agent worker start`、Kubernetes operator、Helm chart 与 fleet management API 这类面向平台落地的运行方式。

2. 为什么值得工程团队关注：这条更新解决的不是“agent 会不会写代码”，而是很多团队真正卡住的上线前提——测试环境、内部依赖、私网端点和合规边界能否被 agent 安全访问。对于已有复杂 CI、内网包仓、受限网络或高合规要求的团队，这是把 coding agent 从 PoC 推进到正式工程流程的关键条件。

3. 最可能影响的环节：开发 / 测试 / 流程 / 选型。

【工程意义】
- 对开发：需要访问内网依赖、私有包源、内部服务或真实预发环境的任务，理论上可以交给 agent 执行，不再只能停留在“公网可跑的 demo 仓库”。
- 对测试：agent 若能进入自有网络，就有机会直接跑更接近真实环境的集成测试、回归测试和内部诊断工具，但也意味着权限边界、密钥注入和环境隔离必须重新评审。
- 对流程：平台团队可以把 agent 运行环境纳入现有基础设施与合规治理，而不是额外维护一套平行的“云端代理特例流程”。

【建议动作】
- 立即可做：盘点当前哪些 agent 场景因“只能在外部云端执行”而受阻，尤其是依赖内网构建缓存、私有依赖、专有测试环境的仓库。
- 本周可验证：选一个非核心仓库做最小试点，验证自托管 worker 能否稳定完成 clone、依赖安装、测试执行、日志回传与结果审计。
- 暂时观察：在把关键仓库接入前，先补权限模型、出网策略、密钥生命周期、审计日志和资源清理策略，不要只验证“能跑起来”。

【来源】
- Cursor Changelog：Self-hosted Cloud Agents（Mar 25, 2026）
- Cursor Blog：Cursor now supports self-hosted cloud agents

### 二、GitHub 开始在使用指标里单列 Copilot coding agent 活跃用户，agent 采用率可进入组织治理面板

1. GitHub 官方 Changelog 页面明确标注为 `Mar.25`，时间边界成立。
2. Copilot usage metrics 新增 `used_copilot_coding_agent` 字段，企业和组织管理员可以在日级和 28 天窗口中识别哪些用户实际触发过 Copilot coding agent 会话。
3. GitHub 明确区分了 IDE agent 模式的 `used_agent` 与云端 Copilot coding agent 的使用情况，且统计覆盖“给 issue 指派 Copilot”与“在 PR 评论里 @copilot”这两类入口。

2. 为什么值得工程团队关注：许多团队此前能看到的是“Copilot 总体是否被用”，但难以判断 coding agent 这种更高权限、更高成本、对流程影响更深的形态是否真的被采用。现在组织终于可以把“谁在用 coding agent、在哪条入口用、采用是否扩散”纳入度量与治理，而不是只靠主观反馈。

3. 最可能影响的环节：开发 / 测试 / 流程。

【工程意义】
- 对开发：团队可以区分“IDE 补全类 Copilot 使用”与“后台代理式改代码”使用，避免把两种完全不同的开发模式混成一个 adoption 指标。
- 对测试：如果组织准备扩大 agent 试点，可以把使用指标与 CI 通过率、回滚率、补测比例等数据联动，看 adoption 是否伴随质量波动。
- 对流程：平台或管理团队终于有了组织级可观测入口，可据此设计试点、灰度、培训和审计策略，而不是默认把所有 Copilot 使用都视为同一种行为。

【建议动作】
- 立即可做：把 `used_copilot_coding_agent` 纳入现有 Copilot 运营报表，至少与仓库类型、团队类型和风险等级做一次交叉分析。
- 本周可验证：选一组试点团队，对比 “只用 IDE Copilot” 与 “实际使用 coding agent” 的 PR 周期、失败测试修复速度和 reviewer 负担变化。
- 暂时观察：单纯使用量不是质量结论，短期内不要把该指标直接当作推广 KPI，应与回归质量和权限治理一起看。

【来源】
- GitHub Changelog：Copilot usage metrics now identify active Copilot coding agent users（Mar.25, 2026）

### 三、GitHub 强化 Copilot for Jira：支持模型选择、Jira→PR 可追踪命名，并可通过 MCP 接入 Confluence 上下文

1. GitHub 官方 Changelog 页面明确标注为 `Mar.25`，时间边界成立。
2. GitHub 为 Copilot coding agent for Jira 补充了更清晰的集成指引与报错说明，并允许用户直接在 Jira 评论中指定本次任务要使用的模型。
3. 新版本会让 agent 在分支名和 PR 标题中带上 Jira ticket 号，并可通过配置 Atlassian MCP server 读取 Confluence 页面，为 Jira 工单执行补充设计文档和规格说明上下文。

2. 为什么值得工程团队关注：这不是单纯“Jira 集成更方便”，而是在补齐真实团队最缺的三件事——可追踪性、上下文完整性和任务级可控性。工单到代码的映射、设计文档到 agent 的上下文注入、以及针对任务切换模型，都会直接影响 agent 在现有研发流程里的可用性。

3. 最可能影响的环节：开发 / 测试 / 流程。

【工程意义】
- 对开发：工单评论区开始变成 coding agent 入口，且能按任务挑模型，适合把修 bug、补实现、按规格落代码等动作前移到工单侧。
- 对测试：如果 agent 能读到 Confluence 规格与 Jira 上下文，团队应同步检查它是否也拿到了测试准入标准，否则可能“按设计做了实现”但没有覆盖验收条件。
- 对流程：Jira ticket 号进入分支名、PR 标题与回链信息后，工单—代码—评审的可追踪性明显增强，更适合纳入审计与交付流程。

【建议动作】
- 立即可做：为试点团队梳理一份最小接入模板，包括 Jira 评论触发规范、可选模型策略，以及 Confluence 页面的最小授权范围。
- 本周可验证：挑一个规格明确、影响面有限的 Jira issue，实测“Jira 评论触发 -> 读取 Confluence -> 开分支/提 PR -> 人工 review”的闭环是否稳定。
- 暂时观察：在未验证文档质量与权限边界前，不要默认把所有 Confluence 空间都开放给 agent；规格陈旧或冲突时，错误上下文会直接放大实现偏差。

【来源】
- GitHub Changelog：GitHub Copilot for Jira — Public preview enhancements（Mar.25, 2026）
- GitHub Docs：Integrate coding agent with Jira / Extending Copilot coding agent with MCP

## 报告观点

### 四、Google 发布 Gemini API developer skill 与 117 条评测结果，给 coding agent 的“文档知识注入”提供了可复用做法

1. Google Developers Blog 官方文章页面标注 `MARCH 25, 2026`，且附带官方 GitHub 仓库，时间边界可成立。
2. Google DeepMind 公布了 `gemini-api-dev` skill，用来向 coding agent 注入 Gemini API 的当前模型、SDK、样例代码和文档入口，并明确建议把文档作为 source of truth。
3. 官方同时给出一个包含 117 个 prompt 的评测框架，测试 Python/TypeScript 代码生成任务；文中披露，加入该 skill 后，Gemini 3 系列在使用旧 SDK/旧模型上的错误显著减少，而旧 2.5 系列收益较弱。

2. 为什么值得工程团队关注：这条信息的价值不在“Google 发布了一个自己的 skill”，而在它给出了一个很实用的工程判断——很多 coding agent 的错误不是推理不会，而是知识过时、SDK 迁移和最佳实践漂移导致的。与其一味加长 prompt，不如把“最新接口知识 + 官方文档入口 + 可回归评测”做成可版本化资产。

3. 最可能影响的环节：开发 / 测试 / 流程 / 选型。

【工程意义】
- 对开发：如果团队维护内部 SDK、平台 API 或脚手架，可以借鉴这种 skill/知识包形态，把最新规范直接注入 agent，减少其继续调用旧接口。
- 对测试：Google 给出的 117 条 prompt eval 说明，知识注入应当配套离线回归，而不是只靠主观感受判断“agent 好像更懂了”。
- 对流程：这类知识资产适合进入版本治理：何时更新、谁审核、如何回滚、哪些文档才是 source of truth，都应制度化。

【建议动作】
- 立即可做：挑一个你们最容易被 agent 调错的内部 SDK 或平台接口，整理最小版“agent 知识包”，至少包括当前接口、禁用旧写法、官方文档入口和示例代码。
- 本周可验证：仿照文中的做法准备一小组高频 prompts，对比注入知识前后的错误类型，重点看是否减少旧 SDK、旧模型、旧配置的误用。
- 暂时观察：如果知识包无法稳定更新，过期 skill 反而会误导 agent；上线前先定义维护责任和版本检查机制。

【来源】
- Google Developers Blog：Closing the knowledge gap with agent skills（MARCH 25, 2026）
- GitHub：google-gemini/gemini-skills
