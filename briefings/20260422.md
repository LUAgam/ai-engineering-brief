## 生成式AI

### 一、OpenAI 在 Responses API 中正式推出 WebSocket 模式：把 agent rollout 整体延迟降低 40%，并把推理速度推高到 1000 TPS

1. OpenAI 官方博客 RSS 中该文 `pubDate` 为 `Wed, 22 Apr 2026 10:00:00 GMT`，折算北京时间 `2026-04-22 18:00`，时间边界成立。
2. 这次不是模型升级，而是 Responses API 传输层重构：用 WebSocket 维持持久连接，复用 `previous_response_id` 在连接内的内存缓存（previous response 对象、tool 定义、已渲染 token 等），让 agent 多回合调用不再重复打全量上下文。
3. 官方公布的可量化指标：Codex 主流量已切到 WebSocket 模式；GPT-5.3-Codex-Spark 在专用 Cerebras 硬件下达到 ~1000 TPS、峰值 4000 TPS；Vercel AI SDK / Cline 多文件工作流 / Cursor 上的 OpenAI 模型分别报告 30%~40% 的延迟下降。

为什么值得工程团队关注：随着推理本身越来越快，过去“被 GPU 卡住”的 agent loop 现在被卡在 API 服务的 CPU 路径和重复处理上。WebSocket 模式把这块结构性开销直接消除，对编码 agent、长链路工具调用 agent 这类“多回合短消息”的工作流来说，是难得的一次工程层级而非模型层级的提速。

最可能影响的环节：开发 / 选型 / 平台底座。

【工程意义】
- 对开发：使用 Responses API 自建 agent 的团队，仅做传输层切换就有显著首字节与端到端时延收益，且无需重写 input/output 形状。
- 对测试：agent 多步任务整体时长缩短意味着批量 eval、回归脚本和长任务实测能用更密的迭代频次跑，eval 成本下降。
- 对流程：选型评估应把“API 传输模式 + 模型 TPS”一起看，单独看 token 价格或单步时延会低估快模型 + 持久连接组合的实际收益。

【建议动作】
- 立即可做：盘点你们现有基于 Responses API 的 agent，标记“多回合 + 短增量上下文”路径，优先在这些路径上启用 WebSocket 模式做 A/B。
- 本周可验证：对 1 个真实 coding agent 任务跑端到端延迟与 TTFT 对比，记录下 30%/40% 的官方数字在你们工作负载上是否成立。
- 暂时观察：WebSocket 长连接对网关、负载均衡、可观测性栈有要求，先确认基础设施是否对长连接友好，再考虑大面积切流。

【来源】
- OpenAI 官方博客：`Speeding up agentic workflows with WebSockets in the Responses API`
- OpenAI 官方 RSS（用于核验时间戳）：`https://openai.com/news/rss.xml`

### 二、OpenAI 推出 Workspace agents：把 GPTs 升级成由 Codex 驱动、可在 Slack/ChatGPT 内共享的“团队级 agent”

1. OpenAI 官方博客 RSS 中该文 `pubDate` 为 `Wed, 22 Apr 2026 10:00:00 GMT`，折算北京时间 `2026-04-22 18:00`，时间边界成立。
2. Workspace agents 由 Codex 提供底层能力，可在云端持续运行，可按计划定时触发，也可部署到 Slack 接收消息；面向 ChatGPT Business / Enterprise / Edu / Teachers 计划，Research Preview 阶段，2026 年 5 月 6 日前免费、之后按 credit 计费。
3. 治理面同步给齐：管理员可控制 agent 可用的连接器和动作、可要求敏感动作（编辑表格、发邮件、加日历）走人工审批；Compliance API 给出每个 agent 的配置、变更和运行记录；内置对外部内容做 prompt injection 防护。

为什么值得工程团队关注：这是 GPTs 之后 OpenAI 第一次把“团队共享 agent”做成 SDLC 化的产品形态——agent 有版本、有 owner、有审批策略、有运行分析、可被组织级管理。对工程组织来说，意味着以后“每个团队的常用工作流”有了一个可托管、可审计的官方载体，而不是各自维护脚本和 GPTs。

最可能影响的环节：流程 / 治理 / 协作工具链。

【工程意义】
- 对开发：研发团队的常见“流程胶水”（PR 自动整理、上线检查、值班报告、跨工具同步）可以用 workspace agent 实现并共享，不必每个团队再造一遍。
- 对测试：QA / 测试效能团队可以把“缺陷分流、回归报告生成、跨系统对账”这类长链路、跨工具任务做成可观测的 agent。
- 对流程：admin 控制 + Compliance API + 审批策略意味着合规口径终于有了官方挂载点，可以替代之前 GPTs 缺乏审计的尴尬。

【建议动作】
- 立即可做：选 1~2 个团队内重复性最高的“跨工具协调”工作流（如周报拉数、合规筛查、Lead 跟进），用 workspace agent 做一个最小可用版本对比脚本方案。
- 本周可验证：在 Slack 频道接入一个内部支持/答疑 agent，记录响应时间、人工接管率和误答率，作为后续企业级 rollout 的基线。
- 暂时观察：5 月 6 日开始按 credit 计费，先把试点期摸清单 agent 的 run 次数、tool call 次数和数据出入口，再决定要不要进入企业广覆盖。

【来源】
- OpenAI 官方博客：`Introducing workspace agents in ChatGPT`

### 三、OpenAI 开源 Privacy Filter：1.5B 参数（50M 激活）的 PII 检测模型，可本地跑、最长 128k 上下文、Apache 2.0

1. OpenAI 官方博客 RSS 中该文 `pubDate` 为 `Wed, 22 Apr 2026 00:00:00 GMT`，折算北京时间 `2026-04-22 08:00`，时间边界成立。
2. Privacy Filter 是一个双向 token 分类 + 约束 Viterbi span 解码的小模型，1.5B 总参数 / 50M 激活；定义了 8 类 PII 标签（包括 `account_number`、`secret` 等覆盖 API key、密码、银行账号），单次前向标注全文，可调精度/召回。
3. 官方在 PII-Masking-300k 上报告 F1 96%（修正标注问题后 97.43%），并强调可本地部署、避免把待脱敏数据发到外部服务；模型权重已在 Hugging Face 与 GitHub 上以 Apache 2.0 开源。

为什么值得工程团队关注：这是少见的“OpenAI 自己用、然后开源、且可商用”的安全基础设施模型。对要做日志脱敏、训练数据清洗、RAG 索引前置过滤、用户对话留存治理的团队来说，过去要么写正则要么调专有 SaaS，现在多了一个能本地跑、效果接近前沿且可微调的工程组件。

最可能影响的环节：测试 / 流程 / 数据治理 / 合规。

【工程意义】
- 对开发：可作为推理前置/后置的 guardrail，用于聊天日志、prompt log、错误堆栈中的 PII 自动脱敏。
- 对测试：对真实生产数据做脱敏后再进 eval、回归与离线分析，能解决“想用真实数据做评测但不能”的长期痛点。
- 对流程：对 RAG 索引、知识库构建、用户反馈分析等环节加入显式的 PII/secret 过滤步骤，从工程层降低数据泄露风险。

【建议动作】
- 立即可做：在你们当前最敏感的一条数据流（推理日志或客服对话）上跑一次基准对比，看 Privacy Filter 与现有正则/SaaS 的召回与误删差距。
- 本周可验证：把 Privacy Filter 嵌入 RAG 入库前的清洗管线，统计被打 `secret` / `account_number` 标签的命中数据量，定位历史泄露风险点。
- 暂时观察：官方明确这不是合规工具、对短文本和小语种存在边界，因此要保留人工 review 通道，不要把它当成最终防线。

【来源】
- OpenAI 官方博客：`Introducing OpenAI Privacy Filter`
- 模型权重：Hugging Face `openai/privacy-filter`、GitHub `openai/privacy-filter`（Apache 2.0）

## 工程工具链

### 四、Google 推出 Agents CLI in Agent Platform：把“为编码 Agent 准备的脚手架命令行”作为产品形态发布

1. Google Developers Blog 文章 `Agents CLI in Agent Platform: create to production in one CLI` 页面日期为 `APRIL 22, 2026`；该博客默认美西时区，对应北京时间 `2026-04-22` 晚间至次日凌晨，按页面日期收录但叙述上保留时区不确定性。
2. Agents CLI 是 Google 面向编码 Agent（明确点名 Gemini CLI / Claude Code / Cursor）的脚手架与部署工具：通过 `uvx google-agents-cli setup` 注入 Skills，让 coding agent 直接调用 `agents-cli create / eval / infra / deploy / publish` 等命令，把项目从本地脚手架到 Agent Runtime / Cloud Run / GKE 的完整链路串起来。
3. 工具支持 Agent Mode 与 Human Mode 两种使用方式：编码 agent 自动调度命令的同时，开发者也可以在终端逐条执行；并提供 `agents-cli eval run / compare` 用于对评测结果做版本对比。

为什么值得工程团队关注：这是“工具链针对 agent 用户做设计”的明显信号——核心目的是降低 coding agent 在组合云资源时的“上下文溢出和 token 浪费”，让 AI 写出来的项目就是“符合 Google Cloud 标准模板、能直接部署”的形状。如果团队已经在用 Gemini / Claude Code / Cursor 做服务级开发，这一类 CLI 会直接影响产出质量。

最可能影响的环节：开发 / 流程 / 选型。

【工程意义】
- 对开发：让 coding agent 输出的项目天然带有 IaC、CI/CD、评测 harness，减少“能写代码但不会部署”的脱节。
- 对测试：CLI 内置 `eval` 子命令并支持运行间对比，让 agent 评测从“一次性脚本”往可比较、可回归的方向走。
- 对流程：从 `create` 到 `deploy` / `publish` 的完整命令栈，是把 agent 从原型推到 Agent Runtime / Cloud Run 的标准路径，对企业内部规范化部署很有参考价值。

【建议动作】
- 立即可做：在一个内部 PoC agent 项目上试跑一次 `agents-cli create` + `agents-cli eval run`，评估默认脚手架与你们现有约定的差距。
- 本周可验证：让 Gemini CLI / Claude Code 在你们的 sandbox 中调用 Agents CLI 完成一个端到端 deploy，记录 token 使用、失败点和需要人工兜底的步骤。
- 暂时观察：GA 时间和 Agent Runtime 之外的目标（GKE、Cloud Run）支持完整度还需观察，深度耦合前先评估锁定风险。

【来源】
- Google Developers Blog：`Agents CLI in Agent Platform: create to production in one CLI`
- GitHub：`google/agents-cli`（文中链接）

### 五、GitHub Copilot 在 VS Code 正式开放 BYOK：Business / Enterprise 用户可接 Anthropic、Gemini、OpenAI、OpenRouter、Azure 与本地 Ollama / Foundry Local

1. GitHub Changelog 页面 URL 为 `2026-04-22-bring-your-own-language-model-key-in-vs-code-now-available`，发布日期 `Apr 22, 2026`；同时还有同日发布的 C++ Language Server for Copilot CLI（公开预览），均落入北京时间 4 月 22 日范围内（按 GitHub 默认 PT 时区估算）。
2. BYOK 在 VS Code Chat（含 plan agent 与自定义 agent）全量可用；不覆盖代码补全；用量直接计入用户在外部供应商处的账单，不消耗 Copilot 配额。策略默认开启，组织管理员可在 Copilot 策略里关掉。
3. 同日 Copilot CLI 还引入了 Microsoft C++ Language Server 的公开预览，弥补了 grep 风格代码搜索在 C++ 复杂宏/模板/构建配置上的不足；推荐在查询里追加 “Use the C++ LSP”。

为什么值得工程团队关注：BYOK 的关键不是“接入更多模型”，而是把“Copilot 入口”和“模型成本/合规归属”解耦——之前用 Copilot 就意味着算到 GitHub 账上、走 GitHub 模型路径，现在企业可以把模型走自己的 Anthropic / Azure / 本地 Ollama，但仍保留 Copilot 在 VS Code 内的工作流和扩展。对要做模型多供应商策略、私有/本地推理、跨厂商对账的团队来说价值明显。

最可能影响的环节：开发 / 选型 / 成本治理。

【工程意义】
- 对开发：在 VS Code 里可直接切到 Anthropic 或本地 Ollama 模型做特定语言/框架的实验，不必离开 Copilot。
- 对测试：可在同一编辑器内对相同任务跑不同模型做主观对比，建立内部“模型 vs 任务”观感数据。
- 对流程：管理员策略层可控制 BYOK 开关，便于把“模型成本”从 GitHub 账单中切出，由各业务线对 API 供应商负责。

【建议动作】
- 立即可做：盘点你们现已购买的模型 key（Anthropic / Azure OpenAI / OpenRouter），评估接入 Copilot VS Code Chat 后能减少多少“为同一模型双重订阅”的浪费。
- 本周可验证：选 1 个 C++ 重仓库，启用 Copilot CLI 的 C++ LSP 预览，比较 grep 模式 vs LSP 模式在“跨文件改动”任务上的修改命中率。
- 暂时观察：BYOK 不计入 Copilot 配额，意味着用量统计和成本归属要重做；先把账单口径和审计流程对齐，再放开给所有团队使用。

【来源】
- GitHub Changelog：`Bring your own language model key in VS Code now available`
- GitHub Changelog：`C++ code intelligence for GitHub Copilot CLI in public preview`
