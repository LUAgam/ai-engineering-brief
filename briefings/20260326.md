## 工程工具链

### 一、GitHub 允许直接让 `@copilot` 处理 PR 合并冲突，agent 开始进入更靠后的交付收尾环节

1. GitHub 官方 Changelog 页面明确标注为 `Mar.26`，时间边界成立。
2. Copilot coding agent 现在支持在 pull request 评论中通过 `@copilot Merge in main and resolve the conflicts` 直接处理 merge conflict。
3. GitHub 明确说明该 agent 仍在其云端开发环境中完成改动，并会检查构建和测试是否通过后再推回结果。

2. 为什么值得工程团队关注：过去很多团队愿意把“补测试、修 review comment”交给 agent，但合并冲突常常仍是人工瓶颈。现在 agent 开始进入更靠近合并前的收尾阶段，意味着 PR 回合数、冲突修复等待时间，以及谁来对最终合并状态负责，都需要重新定义。

3. 最可能影响的环节：开发 / 测试 / 流程。

【工程意义】
- 对开发：低到中复杂度的冲突修复可以从人工切分上下文、手工 rebase，转向“在 PR 评论中直接委托 agent 处理并回推”。
- 对测试：既然 GitHub 强调 agent 会检查 build 和 tests，团队需要确认它跑的是不是与你们合并门禁一致的验证集，而不是只覆盖最小冒烟检查。
- 对流程：冲突解决不再只是开发者本地操作，PR 评论区会进一步成为 agent 协作入口，审批责任、回滚方式和审计留痕都要跟上。

【建议动作】
- 立即可做：限定该能力只用于依赖更新、测试补丁、文档同步等低风险冲突，先不要直接放开到数据库迁移、权限逻辑和跨模块重构。
- 本周可验证：选一个 CI 稳定的仓库，实测“主干推进导致 PR 冲突 -> `@copilot` 解决 -> CI 回归 -> 人工复核”的闭环成功率和返工率。
- 暂时观察：若冲突涉及 generated files、schema 变更或多语言联动文件，先保留人工主导，避免 agent 在局部可编译但全局语义错误的情况下误合并。

【来源】
- GitHub Changelog：Ask @copilot to resolve merge conflicts on pull requests（Mar.26, 2026）
- GitHub Docs：Asking GitHub Copilot to make changes to an existing pull request

### 二、GitHub 在 Actions 运行摘要中直接展示 Agentic Workflow 配置，agent 执行开始具备更好的审计与复盘入口

1. GitHub 官方 Changelog 页面明确标注为 `Mar.26`，时间边界成立。
2. 现在任意 agentic workflow 的 GitHub Actions run summary 中，都可以直接看到该次运行所使用的 Agentic Workflow markdown 配置。
3. GitHub 将这次更新的价值明确描述为减少页面切换，并让使用者能直接看到工作流运行时的精确配置。

2. 为什么值得工程团队关注：agent workflow 的一个常见问题不是“跑没跑”，而是事后很难还原它当时到底按什么配置执行。把配置直接带进 run summary，意味着排障、审计、复盘、跨团队 review 时，不再需要在仓库文件、历史提交和运行结果之间来回比对。

3. 最可能影响的环节：测试 / 流程 / 选型。

【工程意义】
- 对开发：开发者在查看一次 agent 任务失败或异常输出时，可以更快确认问题是出在 prompt/config 还是代码本身。
- 对测试：测试与平台团队更容易把“失败 run 对应的实际 agent 配置”纳入缺陷复盘，减少“代码没变但 agent 行为变了”这类定位成本。
- 对流程：配置可见性提升后，agent workflow 更适合进入审计、变更评审和运行后分析，而不是继续作为黑箱自动化。

【建议动作】
- 立即可做：把 run summary 中的配置截图或链接纳入 agent 故障复盘模板，要求每次异常运行都保留配置上下文。
- 本周可验证：挑一个已有 agentic workflow 的仓库，对比更新前后的排障耗时，重点看值班同学是否能只靠 run summary 还原执行上下文。
- 暂时观察：如果你们把敏感参数写进 markdown 配置，要先复核摘要展示范围与最小暴露原则，避免“可观测性增强”变成新泄漏面。

【来源】
- GitHub Changelog：View Agentic Workflow configs in the Actions run summary（Mar.26, 2026）
- GitHub Agentic Workflows Documentation

### 三、GitHub-hosted runners 的自定义镜像正式 GA，CI 环境标准化与预装依赖开始从预览走向正式能力

1. GitHub 官方 Changelog 页面明确标注为 `Mar.26`，时间边界成立。
2. Custom images for GitHub-hosted runners 在 2025 年 10 月公开预览后，于 3 月 26 日正式 GA。
3. GitHub 明确强调，这项能力允许团队从官方基础镜像出发，预装工具、依赖、证书和配置，以获得更快、更一致、更安全的工作流执行环境。

2. 为什么值得工程团队关注：很多 AI 工程团队的 CI/CD 问题并不在工作流语法，而在执行环境不稳定——临时安装依赖慢、证书和私有工具链配置分散、不同仓库环境不一致。自定义镜像 GA 后，标准化 runner 环境终于能作为正式平台能力推进，而不是依赖脚本层临时拼装。

3. 最可能影响的环节：开发 / 测试 / 流程。

【工程意义】
- 对开发：预装常用 SDK、构建工具和安全证书后，开发者触发的 CI 等待时间有机会明显下降，且环境差异更少。
- 对测试：回归测试、E2E 测试和需要浏览器/系统依赖的任务更容易获得稳定环境，减少“测试逻辑没变、底层镜像漂移导致失败”的噪音。
- 对流程：平台团队可以把 runner 镜像版本、补丁更新、证书注入和工具链审批纳入集中治理，降低仓库级重复维护。

【建议动作】
- 立即可做：盘点哪些仓库在 workflow 里重复安装大型依赖、内部证书或专有 CLI，把这些候选项整理成第一版基础镜像需求。
- 本周可验证：挑一类耗时长且环境敏感的工作流，对比“现装依赖”与“自定义镜像预装”在时长、失败率和可重现性上的差异。
- 暂时观察：不要急于为每个团队各做一套镜像，先用少量标准基线镜像覆盖大多数用例，否则镜像治理本身会成为新的运维负担。

【来源】
- GitHub Changelog：Custom images for GitHub-hosted runners are now generally available（Mar.26, 2026）
- GitHub Docs：Using custom images / Managing larger runners

## 生成式AI

### 四、GitHub 的凭证吊销 API 新增对 OAuth 与 GitHub App 凭证支持，泄漏处置可以更快进入自动化流程

1. GitHub 官方 Changelog 页面明确标注为 `Mar.26`，时间边界成立。
2. Credential revocation API 新增支持 OAuth app tokens、GitHub App user-to-server tokens 与 GitHub App refresh tokens，同时保留对 classic 与 fine-grained PAT 的支持。
3. GitHub 明确说明该 API 可匿名调用、支持批量提交，命中有效 token 后会自动吊销，并在 token owner 的 security log 中记录事件；组织访问权限也会被立即移除。

2. 为什么值得工程团队关注：随着 coding agent、CI 工作流和各种平台集成增多，真正高频暴露的往往不是用户 PAT，而是 OAuth 与 GitHub App 这类自动化凭证。现在吊销 API 覆盖范围扩大，意味着你们可以把“发现泄漏 -> 自动吊销 -> 通知负责人 -> 重新签发”做成更完整的自动化响应链路。

3. 最可能影响的环节：测试 / 流程 / 安全选型。

【工程意义】
- 对开发：一旦开发者在仓库、日志或脚本中误暴露凭证，平台侧可以更快触发统一处置，而不必依赖各系统手工登录逐个撤销。
- 对测试：安全演练可以更贴近真实场景，覆盖 GitHub App 与 OAuth token 泄漏后的批量吊销、权限移除和恢复流程。
- 对流程：该能力适合接入 secret scanning、日志告警或内部安全机器人，把响应时间从人工小时级压缩到自动化分钟级。

【建议动作】
- 立即可做：梳理现有泄漏应急脚本是否只覆盖 PAT；若是，补上 OAuth 和 GitHub App 凭证类型映射与处置路径。
- 本周可验证：做一次桌面推演或沙箱演练，验证从发现暴露 token 到调用吊销 API、通知 owner、补发凭证的端到端链路是否通畅。
- 暂时观察：该 API 有速率与单次请求数量限制，正式接入大规模扫描系统前要先评估分批策略、误报处理和幂等设计。

【来源】
- GitHub Changelog：Credential revocation API now supports GitHub OAuth and GitHub app credentials（Mar.26, 2026）
- GitHub Docs：Credential revocation API / Token expiration and revocation
