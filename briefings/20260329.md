## 工程工具链

### 一、Claude Code v2.1.87 在北京时间 3 月 29 日上午发布，修复 Cowork Dispatch 消息未送达

1. Anthropic 官方 GitHub Release `v2.1.87` 的 `published_at` 为 `2026-03-29T02:17:02Z`，折算北京时间为 2026-03-29 10:17:02，时间边界成立。
2. 官方 release note 只有一条核心变更：**Fixed messages in Cowork Dispatch not getting delivered**，指向的是 Claude Code 与 Dispatch / Remote 场景中的消息投递可靠性问题。
3. Anthropic 官方文档中，Claude Code Desktop 与 Remote Control 已把 Dispatch 作为跨设备发起任务、继续会话的入口之一；因此这次修复虽小，但影响的是“任务能否被可靠送达并接续执行”这个基础链路。

2. 为什么值得工程团队关注：如果团队已经把 Claude Code 用于移动端发起任务、桌面端接力执行、值班场景远程处理，消息丢失不是体验问题，而是流程中断问题。对把 agent 纳入真实开发流程的团队，任务投递链路的可靠性优先级高于新功能。

3. 最可能影响的环节：开发 / 流程。

【工程意义】
- 对开发：依赖跨设备继续会话、远程派发任务的成员，任务落地率会更高，减少“看起来已发出、实际上未执行”的误判。
- 对测试：需要补 Dispatch/Remote Control 场景回归，重点验证移动端发起、桌面端接收、长会话续接与失败重试。
- 对流程：值班、异步协作、跨时区任务派发等流程，需要把“消息是否成功送达”纳入 agent 基础健康检查，而不能只盯模型响应成功率。

【建议动作】
- 立即可做：如果团队在用 Claude Code Desktop / Remote Control / Dispatch，补一轮最小验证，确认跨设备派发任务能稳定落到目标会话。
- 本周可验证：把“远程发起任务后是否真正进入执行队列”加入回归脚本或冒烟清单，尤其覆盖弱网、前后台切换和长会话恢复。
- 暂时观察：这是 patch 级修复，但 release note 信息有限；在大规模推广前，继续观察后续 release 是否补充更多与 Dispatch 相关的稳定性修复。

【来源】
- GitHub Releases：anthropics/claude-code `v2.1.87`
- Anthropic 官方文档：Claude Code / Remote Control 相关说明

### 二、OpenClaw 2026.3.28 在北京时间 3 月 29 日发布稳定版，插件审批、CLI backend 与 Codex/Claude/Gemini 接入进一步统一

1. OpenClaw 官方 GitHub Release `v2026.3.28` 的 `published_at` 为 `2026-03-29T01:34:30Z`，折算北京时间为 2026-03-29 09:34:30，时间边界成立。
2. 这次 release note 信息完整，且直接覆盖 agent 工程核心面：新增插件 `before_tool_call` 异步 `requireApproval`、将 Claude CLI / Codex CLI / Gemini CLI 的 backend 与默认推理入口进一步迁到插件表面、默认为 OpenAI 与 OpenAI Codex 模型启用 `apply_patch`，并补上 MCP/channels、sandbox、failover、CLI/MCP 隔离等大量修复。
3. 其中几条最值得工程团队注意：一是插件可以在工具执行前暂停并请求人工批准；二是 bundled CLI backend 自动加载更激进，减少显式 `plugins.allow` 配置；三是后台 Claude CLI 运行时会强制传入生成的 `--mcp-config` overlay，避免继承用户全局 MCP 服务器。

2. 为什么值得工程团队关注：这不是单点功能更新，而是在把“多模型、多 CLI、多插件、多渠道”的 agent 运行时往统一治理面收拢。审批、插件自动加载、MCP 隔离和 sandbox 权限这些变更，会直接影响团队如何设计默认配置、如何做最小权限、以及如何防止本机环境污染后台 agent 任务。

3. 最可能影响的环节：开发 / 测试 / 流程 / 安全选型。

【工程意义】
- 对开发：Claude CLI、Codex CLI、Gemini CLI 接入路径更统一，减少不同 provider/backend 的配置分叉；`apply_patch` 默认启用也会改变部分代码修改路径。
- 对测试：需要重点回归插件审批流、CLI backend 自动加载、MCP overlay 隔离、sandbox 允许/拒绝规则，以及模型 failover 行为。
- 对流程：平台团队可以把插件审批与 bundled backend 当成标准治理面，统一定义哪些工具必须人工确认、哪些 MCP 服务器允许进入后台任务。

【建议动作】
- 立即可做：检查团队的 OpenClaw 配置是否依赖手工 `plugins.allow`、旧 Qwen portal 认证、或默认继承用户全局 MCP 配置；若有，按新版 release note 调整。
- 本周可验证：设计一组 focused regression，覆盖 `before_tool_call` 审批、Codex/Claude/Gemini 三类 CLI backend、显式禁用/允许工具、后台任务的 MCP 继承边界。
- 暂时观察：新版本改动面很大，且包含 breaking changes；不要直接全量切换生产或团队默认模板，先在多 provider、多 channel、多插件仓库里灰度。

【来源】
- GitHub Releases：openclaw/openclaw `v2026.3.28`
- 官方 release note / compare 说明

## 社区动态

### 三、Orca v1.0.61 在北京时间 3 月 29 日发布，多 agent IDE 补齐 diff、文件导航与终端空闲态提示

1. `stablyai/orca` 官方 GitHub Release `v1.0.61` 的 `published_at` 为 `2026-03-29T00:20:38Z`，折算北京时间为 2026-03-29 08:20:38，时间边界成立。
2. 该版本 release note 明确列出多项直接面向工程工作台的改进：终端支持拖拽 source control 文件、设置页展示快捷键映射、编辑器 header 与文件标签显示 diff 指示、diff 中预览图片文件、快速打开文件导航，以及通过终端标题检测 agent idle 以触发未读提醒。
3. Orca 的定位是跨 worktree、跨终端管理多种 coding agent 的桌面工作台；因此这批改动的价值不在“模型能力”，而在多 agent 并行时的人机协作可见性和操作摩擦。

2. 为什么值得工程团队关注：当团队不只跑一个 agent，而是在多个终端、多个分支、多个 worktree 间切换时，真正拖慢效率的往往不是生成质量，而是“现在谁改了什么、哪一个会话空闲了、哪些文件已提交/未提交”。Orca 这类工作台的细节补强，适合给正在评估多 agent 桌面编排的团队做参考。

3. 最可能影响的环节：开发 / 流程 / 选型。

【工程意义】
- 对开发：diff 指示、图片 diff 预览、快速开文件和拖拽 source control 文件，会降低在多终端间定位修改结果的成本。
- 对测试：适合验证多 worktree、多 agent 并行时的状态同步是否准确，尤其是 source control tab、PR 状态检查与终端空闲提醒。
- 对流程：如果团队在评估“编排层”而不是单一 CLI，本次 release 提供了一个较清晰的工作台能力清单，可反推你们真正需要的可见性功能。

【建议动作】
- 立即可做：如果团队正在试多 agent workbench，列出你们最痛的 5 个状态可见性问题，和 Orca 这版提供的 diff/file/idle 能力逐项对照。
- 本周可验证：在多 worktree、多终端的真实仓库里试跑一次，从“任务开始到提交 PR”全流程观察文件状态、已提交文件展示、空闲提醒是否足够支撑协作。
- 暂时观察：Orca 发布频率较高，同日还有后续 patch；如果要做正式选型，优先观察相邻几个版本是否持续稳定，而不是只看单次功能列表。

【来源】
- GitHub Releases：stablyai/orca `v1.0.61`
- 官方 release note / compare 说明
