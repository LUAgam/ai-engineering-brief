## 工程工具链

### 一、Anthropic 在北京时间 4 月 2 日发布 Claude Code v2.1.90，继续补强恢复链路、权限边界与 PowerShell 安全检查

1. `anthropics/claude-code` 官方 GitHub Release `v2.1.90` 的 `published_at` 为 `2026-04-01T23:41:31Z`，折算北京时间为 `2026-04-02 07:41:31`，时间边界成立。
2. 这一版新增点不多，但关键修复都落在真实生产链路：修复 `--resume` 在 deferred tools、MCP servers、custom agents 场景下首轮请求触发完整 prompt-cache miss 的回归；修复 `Edit`/`Write` 在 `PostToolUse` format-on-save 改写文件后误报 “File content has changed”；修复 auto mode 在“不要 push”“先等 X 再做 Y”这类显式用户边界下仍可能越界执行；并加固 PowerShell 工具权限检查，覆盖尾随 `&` 后台任务绕过、`-ErrorAction Break` 调试挂起、archive extraction TOCTOU 和解析失败后的 deny-rule 降级。
3. 同版还补了长会话与大流式响应的性能问题：SSE transport 对大 frame 改为线性处理、SDK 长会话 transcript writes 不再二次方变慢、`/resume` all-projects 并行加载会话，说明重点仍是让 agent 在真实仓库和长生命周期会话里更稳而不是继续堆入口功能。

2. 为什么值得工程团队关注：这版最有价值的不是新命令，而是把“恢复后是否失去缓存”“格式化器是否打断自动编辑”“agent 是否会违背明确操作边界”“Windows/PowerShell 权限规则是否可绕过”这几类高风险问题继续往可控方向收敛。对已经把 Claude Code 放进半自动开发流、脚本流或企业安全边界里的团队，这些修复直接影响是否敢扩大使用面。

3. 最可能影响的环节：开发 / 测试 / 流程 / 安全治理。

【工程意义】
- 对开发：`--resume`、format-on-save、长会话流式响应这些高频场景更稳，减少“恢复后变慢”“保存后编辑冲突”“会话越跑越卡”的真实阻塞。
- 对测试：应补 focused regression，覆盖 deferred tools + resume、MCP/custom agent 首轮缓存命中、format-on-save 改写后的连续编辑、PowerShell 权限绕过和显式用户边界约束。
- 对流程：如果团队在推动 headless/半自动编码流程，这版提供了更强信号：审批和策略层不能只看 allowlist，还要验证“恢复”“重试”“格式化副作用”“显式指令边界”这些流程细节。

【建议动作】
- 立即可做：把 “不要 push / 先等审批 / 先出计划” 这类显式边界加入你们的 agent 回归用例，确认工具不会因为 auto mode 而跨线执行。
- 本周可验证：用真实仓库跑一轮包含 format-on-save、MCP、自定义 agent、长会话恢复的回归，观察 prompt cache、编辑冲突和 PowerShell 权限提示是否符合预期。
- 暂时观察：继续观察 `v2.1.91+` 是否沿着 resume、权限和长会话性能继续收敛；如果团队有大量 Windows/PowerShell 用户，应优先灰度验证。

【来源】
- GitHub Releases：anthropics/claude-code `v2.1.90`

### 二、Orca 在北京时间 4 月 2 日发布 v1.0.72，补 worktree 外 git 操作、PDF diff 预览与分支名复用风险

1. `stablyai/orca` 官方 GitHub Release `v1.0.72` 的 `published_at` 为 `2026-04-02T00:41:02Z`，折算北京时间为 `2026-04-02 08:41:02`，时间边界成立。
2. 本版 release note 给出的几项改动都很贴近真实工程仓库：**prevent branch name reuse across PR history**，防止 PR 历史里分支名复用；**allow git operations on worktrees outside repo/workspace roots**，允许在 repo/workspace root 之外的 worktree 执行 git 操作；同时新增 **PDF file preview support in diff viewer** 和 “copy remote URL to line number context menu”，补齐 diff 与上下文分享链路。
3. 这些变化说明 Orca 关注点不是单纯聊天体验，而是在往多工作树、多 PR、多制品文件的工程工作台形态靠。尤其是 worktree outside roots 支持，直接对应很多团队把大仓、并行任务或临时修复放在独立 worktree 的习惯。

2. 为什么值得工程团队关注：AI 工作台要真进入日常流程，不能只在单仓根目录、纯源码 diff 里工作。worktree 外 git 操作与 PDF diff 预览，分别对应并行开发和非代码制品审阅这两个真实痛点；分支名复用防护则直接减少“误把历史上下文当当前任务”的流程事故。

3. 最可能影响的环节：开发 / 测试 / 流程 / 选型。

【工程意义】
- 对开发：使用多个 worktree 并行修 bug、试验实现或代理任务时，不必把工作目录强行绑回单一 workspace root。
- 对测试：应补 worktree outside roots、分支名复用、PDF diff 渲染、远程 URL 拷贝等场景的回归，特别是跨平台路径与 Git 状态解析。
- 对流程：评估 AI 工作台类工具时，应把 worktree 兼容、非代码制品 diff、PR 历史隔离作为基础验收项，而不是只看模型回答质量。

【建议动作】
- 立即可做：把团队常见的 worktree 布局、临时修复分支命名和 PDF 规格文档/审计报告 diff，纳入 AI 工具试点清单。
- 本周可验证：构造 repo root 外 worktree 的真实仓库样本，确认 git 操作、diff 展示、远程 URL 分享是否稳定。
- 暂时观察：继续观察 Orca 后续版本是否持续补工作树、diff viewer 和 PR 上下文隔离；如果是，说明它在往团队工作台而非单机聊天工具演进。

【来源】
- GitHub Releases：stablyai/orca `v1.0.72`

### 三、Google Gemini CLI 在北京时间 4 月 2 日更新 nightly 预发布，重点继续收敛 sandbox、plan mode 与 subagent 稳定性

1. `google-gemini/gemini-cli` 官方 GitHub Release `v0.36.0-nightly.20260402.13ccc1645` 的 `published_at` 为 `2026-04-02T04:33:43Z`，折算北京时间为 `2026-04-02 12:33:43`，时间边界成立；但该版本为 **nightly 预发布**，结论应按预发布口径理解。
2. 公开 release note 显示，这个 nightly 的工程变化高度集中在 agent 运行底座：sandbox 方面新增/推进 Linux、macOS、Windows 动态扩展、forbiddenPaths、环境文件 secret visibility lockdown、网络访问和 `__read/__write` 支持；plan mode 方面补了 untrusted folders、plan file 创建死锁、sandbox path resolution、plan mode policy 放宽与稳定化；subagent / agent history / event-driven context 管理也有一批修复和重构。
3. 同版还补了多项与可运营性直接相关的问题：shell 输出 10MB 上限防止 RangeError crash、remove shell outputChunks buffer caching 防止内存膨胀、headless 环境 TTY hang 修复、browser agent 域名限制绕过修复、工具取消后 agent 停止条件修复，以及多项 eval / CI skill / reliability harvester 相关工作。

2. 为什么值得工程团队关注：虽然是 nightly，但它给出的信号很明确——Google 正在把 Gemini CLI 的重点放在 sandbox/plan/subagent 这三条工程主线，而不是只做模型层更新。对于正在评估本地/CLI agent 的团队，这些变化比“又多一个命令”更接近真实落地门槛。

3. 最可能影响的环节：开发 / 测试 / 流程 / 安全治理 / 选型。

【工程意义】
- 对开发：plan mode 在不受信目录、受限 sandbox、长会话和并行子代理里的可用性继续提升，减少“能规划但落不了地”或“能落地但权限策略崩掉”的问题。
- 对测试：nightly 值得重点验证 sandbox 边界、forbiddenPaths、env 文件可见性、plan file 创建、headless TTY、browser allowedDomains、subagent 历史完整性和大输出崩溃保护。
- 对流程：如果团队在做开源 CLI agent 选型，这版说明 Gemini CLI 已明显进入“权限与稳定性工程化”阶段，但仍应放在预发布灰度池中而非直接全员推广。

【建议动作】
- 立即可做：把 sandbox、plan mode、subagent、browser domain restriction 四类场景列为 Gemini CLI 的必测项，不要只做简单命令问答。
- 本周可验证：选一到两个真实仓库，在 nightly 版本上做 worktree/受限目录/headless/大输出/多代理回归，重点看死锁、崩溃和权限逃逸是否收敛。
- 暂时观察：由于是 nightly，先观察其后续 stable 版本是否保留这些修复；若关键能力进入稳定版，再考虑扩大试点。

【来源】
- GitHub Releases：google-gemini/gemini-cli `v0.36.0-nightly.20260402.13ccc1645`

## 生成式AI

### 四、OpenAI 在北京时间 4 月 2 日宣布 Codex 团队版按量付费，编码 agent 试点成本从“席位前置”改为“用量可归因”

1. OpenAI 官方公告《Codex now offers pay-as-you-go pricing for teams》页面日期为 `April 2, 2026`；OpenAI 站点未在正文中给出更精确时区/时间戳，因此时间判断只能按官方页面日期与站点发布口径成立，边界清晰度低于带 UTC 时间戳的 GitHub Release。
2. 公告的核心不是模型能力，而是部署与试点方式变化：ChatGPT Business / Enterprise 工作区现在可添加 **Codex-only seats**，无固定席位费、按 token 用量计费且无 rate limits；同时标准 ChatGPT Business 年价从 `25` 美元/席位下调到 `20` 美元/席位；官方还明确把 Plugins、Automations 和 Codex app 作为团队接入现有系统的主路径。
3. 这意味着团队采用 Codex 的进入方式从“先买整席位”转成“先按工作流与 token 消耗做小范围试点”，成本归因、预算拆分和 ROI 试验都更容易；对平台团队而言，这比单次产品宣传更接近真实采购和推广问题。

2. 为什么值得工程团队关注：很多编码 agent 项目的真正阻力不是能力本身，而是预算模型、试点门槛和用量归因。OpenAI 这次调整直接改变了团队引入 Codex 的试错方式：可以先围绕高价值工作流做有限人群试点，再决定是否扩大到更广泛的 ChatGPT 席位。

3. 最可能影响的环节：开发 / 流程 / 选型 / 成本治理。

【工程意义】
- 对开发：更适合先在高频编码、修复、自动化脚本等具体工作流中做小规模试点，而不是全员一刀切开通。
- 对测试：应把 token 成本、并发请求量、插件/自动化接入成功率、实际任务成功率一起纳入试点评估，而不是只看生成质量。
- 对流程：平台团队可以把 Codex 视作“按工作流计费的工程工具”，重新设计预算归集、成本标签、试点范围和灰度准入。

【建议动作】
- 立即可做：梳理 2~3 个最适合按量试点的高频工程流程，例如代码修复、脚本生成、PR 辅助或内部工具自动化，并为这些流程单独建成本看板。
- 本周可验证：以最小团队规模试用 Codex-only seats，记录 token 消耗、成功率、失败重试率、与现有 IDE/CLI/自动化链路的集成成本。
- 暂时观察：继续观察 OpenAI 后续是否公开更多团队治理、权限控制、审计和自动化运维能力；若只有计费变化而缺少治理能力，扩面要谨慎。

【来源】
- OpenAI 官方公告：`Codex now offers pay-as-you-go pricing for teams`
