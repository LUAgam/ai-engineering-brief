### 一、OpenAI 在 3 月 31 日为 Codex CLI 0.118.0 新增官方 changelog，重点补沙箱网络控制、动态凭证与 MCP/TUI 稳定性

1. OpenAI Developers 官方 `Codex Changelog` 在 `2026-03-31` 新增 `Codex CLI 0.118.0` 条目，属于该日官方 changelog 新增进展；核心新能力包括 Windows 沙箱支持“仅代理出网”的 OS 级出口规则、`codex exec` 支持“命令行 prompt + stdin 管道”组合，以及自定义模型提供方支持动态刷新短期 bearer token。
2. 同版修复集中落在运行时安全与可靠性：首次创建项目本地 `.codex` 文件时也会走正常审批保护；Linux 沙箱重新可靠识别受信任的 `bwrap`；本地 MCP 服务拥有更长启动窗口，失败握手会重新在 TUI 中暴露告警而不是“假成功”。
3. 面向日常操作流的回归修补也很集中：app-server 驱动的 TUI 恢复了 hook 通知回放、`/copy`、`/resume <name>`、`/agent` 线程列表和 skills picker 翻页等关键工作流，减少多会话、多技能、多工具场景下的假死和误判。

2. 为什么值得工程团队关注：这不是单纯的 CLI 小版本迭代，而是在继续把编码 agent 从“能跑”推向“可控”。OS 级网络出口约束、动态 bearer token、MCP 启动告警恢复，分别对应最小权限、企业认证接入、以及工具运行时可诊断性，都是团队把 agent 放进真实仓库和 CI/CD 辅助流程前必须补齐的基础能力。

3. 最可能影响的环节：开发 / 测试 / 流程 / 安全治理。

【工程意义】
- 对开发：`codex exec` 的 prompt+stdin 组合更适合把静态输入、模板 prompt 和 shell pipeline 接起来；动态 bearer token 则降低了自建模型网关或短期凭证环境里的接入摩擦。
- 对测试：应补覆盖 Windows 沙箱代理出网、Linux `bwrap` 解析、多 MCP 启动超时、TUI `/resume` 与 hooks 回放的 focused regression，避免“功能存在但运维不可用”。
- 对流程：平台/安全团队可以把“仅代理出网”“首次写入审批”“MCP 启动告警可见”视为 agent 落地门槛，而不是把权限和诊断留给使用者手工兜底。

【建议动作】
- 立即可做：盘点团队内所有 Codex/类似 CLI agent 的出网策略、短期凭证刷新方式、首次写入审批链路，确认是否仍依赖环境变量级别的软控制。
- 本周可验证：做一组最小回归，覆盖 Windows 受限沙箱、Linux `bwrap` 路径、多本地 MCP server 启动、`/resume`、`/copy`、hook replay 和 bearer token 轮换。
- 暂时观察：继续观察 0.118.x 后续补丁是否围绕同一批运行时问题收敛；如果团队主要在 macOS 交互式场景使用，可先小范围升级验证。

【来源】
- OpenAI Developers：Codex Changelog（2026-03-31 / Codex CLI 0.118.0）

### 二、Microsoft 在北京时间 3 月 31 日发布 Copilot Chat 预览版 v0.43.2026033101，把工作区元数据、会话连接与 agent 调试继续做实

1. `microsoft/vscode-copilot-chat` 官方 GitHub Release `v0.43.2026033101` 的 `published_at` 为 `2026-03-31T00:44:33Z`，折算北京时间为 `2026-03-31 08:44:33`，时间边界成立。
2. 该版最关键的新增不是 UI，而是可观测性与运行时细节：把 git branch、commit、remote、file path 等工作区元数据加入 OTel 事件与 GitHub telemetry；会话侧改为“一次对话一个 WebSocket”；并新增/补强 Copilot CLI Controller API、session context key、folder session 的 repositoryPath。
3. 修复项同样偏工程底座：后台 compaction 在工具调用间正确生效、`FolderRepositoryManager` 在取 modified files 前先清工作区变更状态、coding agent 启动时使用当前选中模型、优先采用 upstream base branch，并限制 discard changes 动作展示时机。

2. 为什么值得工程团队关注：对于已经把 Copilot Chat 当成 IDE 内 agent 运行时的团队，这一版的价值在于“把对话和工作区绑定得更可观测、更可调试”。工作区元数据进遥测、会话连接模型调整、后台 compaction 修正，直接影响长会话排障、编辑行为归因，以及 PR/分支上下文是否可靠。

3. 最可能影响的环节：开发 / 测试 / 流程。

【工程意义】
- 对开发：长会话和多仓库工作区下，agent 更容易拿到与当前分支、文件、远端相关的上下文信号，减少“模型不知道自己在哪个仓库态”的隐性错误。
- 对测试：需要补回归覆盖 OTel 事件字段完整性、单会话 WebSocket 稳定性、后台 compaction 跨 tool call 行为、selected model 透传，以及 upstream base branch 选择逻辑。
- 对流程：平台团队如果在做 Copilot 使用审计或内部可观测接入，这版提供了更适合建立仓库级/会话级指标的信号面。

【建议动作】
- 立即可做：检查你们当前的 Copilot Chat 观测面是否已经采集工作区/分支/文件级元数据；如果没有，可以把它列为后续接入优先级。
- 本周可验证：用真实仓库做一轮长会话回归，重点看 compaction、分支识别、PR 基线分支、folder session、以及单 WebSocket 模式下的断连恢复。
- 暂时观察：这是预览版，不宜直接定义为团队长期基线；建议先让高频使用者灰度验证，再决定是否扩展到更大人群。

【来源】
- GitHub Releases：microsoft/vscode-copilot-chat `v0.43.2026033101`

### 三、Google 官方在 3 月 31 日发布 ADK Go 1.0，重点把多 Agent 框架补到可观测、可插拔、可审批

1. Google Developers Blog 官方文章《ADK Go 1.0 Arrives!》页面标注日期为 `MARCH 31, 2026`，属于该日官方发布；1.0 版明确新增 OpenTelemetry 集成、插件系统、Human-in-the-Loop 请求确认，以及通过 YAML 定义 Agent。
2. OTel 集成允许开发者把模型调用、工具执行循环接入结构化 trace；插件系统把 logging、安全过滤、Retry and Reflect 这类横切能力从业务 agent 指令里剥离出来；`RequireConfirmation` 则把敏感工具执行前的人类确认做成框架级流程。
3. 同版还强调跨语言与多 Agent 互通：A2A 协议支持 Go/Java/Python agent 通讯，YAML 配置让 agent persona、sub-agent 拓扑和工具声明更容易在不同环境复用，而不必每次都改 Go 代码重编译。

2. 为什么值得工程团队关注：ADK Go 1.0 的价值不在“又一个 agent 框架”，而在它把生产环境最难补的四件事放到了框架默认面：trace、插件化治理、人工审批、配置与代码分离。对已有 Go 服务栈、想把 agent 编排纳入正式后端服务的团队，这比纯 prompt/工具封装更接近可上线状态。

3. 最可能影响的环节：开发 / 测试 / 流程 / 选型。

【工程意义】
- 对开发：Go 团队可以在不重写全部服务基础设施的前提下，把 tracing、plugins、YAML config 和 A2A 协作接进现有工程体系。
- 对测试：应重点验证插件链路顺序、Retry and Reflect 的确定性、敏感工具的 HITL 审批流、以及 YAML 配置与代码定义之间是否存在行为漂移。
- 对流程：这类“框架原生治理能力”意味着平台团队可以把审批、观测、插件策略前置到基座层，而不是在每个 agent 项目里重复造轮子。

【建议动作】
- 立即可做：如果团队主要用 Go 构建内部服务，可先评估是否需要一套统一的 agent tracing、审批和插件治理骨架，而不是继续在业务服务里零散拼接。
- 本周可验证：做一个最小 PoC，覆盖一个带工具调用的 Go agent，接 OTel、挂一个 logging/retry 插件，并对敏感操作启用 `RequireConfirmation`。
- 暂时观察：ADK Go 1.0 是方向明确的首个正式版，但仍需观察其与现有 Go 服务框架、CI、部署链和多语言 agent 体系的磨合成本。

【来源】
- Google Developers Blog：ADK Go 1.0 Arrives!

### 四、Playwright MCP v0.0.69 在北京时间 3 月 31 日发布，把离线网络调试、请求抓取与交互控制继续向测试场景收拢

1. `microsoft/playwright-mcp` 官方 GitHub Release `v0.0.69` 的 `published_at` 为 `2026-03-30T23:56:25Z`，折算北京时间为 `2026-03-31 07:56:25`，时间边界成立。
2. 这一版新增两个直接面向测试的工具：`browser_network_state_set` 可切换离线模式以测试断网/弱网场景；`browser_video_chapter` 可在录屏中插入章节标记，便于回放定位自动化流程节点。
3. 工具增强和修复同样偏回归与定位：`browser_mouse_click_xy` 新增 `button` / `clickCount` / `delay`；`browser_network_requests` 支持过滤与可选 headers/body；工具开始接受普通 CSS/text selector；同时修掉 iframe 内 verify 失效、锁文件隔离、右键非确定性、header 冒号解析、persistent context 隔离错误、存储状态路径校验等问题。

2. 为什么值得工程团队关注：对把浏览器自动化能力暴露给 LLM agent 或测试助手的团队，最缺的往往不是“更多操作”，而是“更容易复现网络故障、更容易抓请求、更容易回看执行片段”。这版正好把网络、录屏、交互和若干非确定性 bug 一起向测试落地收拢。

3. 最可能影响的环节：测试 / 开发 / 流程。

【工程意义】
- 对开发：如果你们在做带浏览器工具的 agent，普通 selector 支持和更细的点击参数能减少额外包装层，缩短从脚本到工具调用的映射。
- 对测试：离线模式、请求过滤、headers/body 抓取、视频章节标记，直接提升 E2E 调试、回归复现和故障归因效率；多项 bug fix 也有助于减少脆弱用例。
- 对流程：测试平台可以把“网络状态切换 + 请求抓取 + 录屏章节”组合成标准排障模板，降低 agent 化 UI 测试的复现实验成本。

【建议动作】
- 立即可做：如果团队在用 Playwright MCP 或同类浏览器 MCP，优先把离线模式、请求过滤和视频章节标记纳入常用调试手册。
- 本周可验证：补一组 focused regression，覆盖离线登录、断网恢复、请求头/body 采集、iframe verify、右键菜单、persistent context 和存储状态导入。
- 暂时观察：MCP 工具面仍在快速扩展，建议把新增工具先纳入测试辅助而非关键生产链路，观察后续版本对非确定性问题的继续修补情况。

【来源】
- GitHub Releases：microsoft/playwright-mcp `v0.0.69`
