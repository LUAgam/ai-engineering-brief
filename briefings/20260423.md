## 生成式AI

### 一、OpenAI 发布 GPT-5.5 / GPT-5.5 Pro：长链路编码与计算机使用全面刷新，Codex 主推"更聪明且更省 token"

1. OpenAI 官方页面 `Introducing GPT-5.5` 与 `GPT-5.5 System Card`、`GPT-5.5 Bio Bug Bounty` 三篇均署 `April 23, 2026`，按美西时区折算大致落在北京时间 4 月 23 日晚到 4 月 24 日凌晨；今日已通过 ChatGPT Plus / Pro / Business / Enterprise 与 Codex 全量推送，API 仅明确"very soon"。
2. 编码与 agent 类 eval 大幅前进：Terminal-Bench 2.0 82.7%（GPT-5.4 75.1%）、Expert-SWE 内部 73.1%、SWE-Bench Pro 58.6%、OSWorld-Verified 78.7%、Tau2-bench Telecom 98.0%（无 prompt 调优）；长上下文方面 Graphwalks BFS 1M F1 从 9.4% 跃到 45.4%、MRCR v2 8-needle 512K-1M 从 36.6% 跃到 74.0%。
3. 工程口径变化最大的是 token 效率与服务延迟：官方明确 GPT-5.5 在 Codex 上"用更少 token 完成同样任务"，且在 NVIDIA GB200/GB300 NVL72 上服务时单 token 延迟与 GPT-5.4 持平；API 定价 5 美元/1M 输入、30 美元/1M 输出，1M context；Codex Fast 模式以 2.5× 价格换 1.5× 速度。

为什么值得工程团队关注：这次更新的真正落点不是某个 benchmark 数字，而是"长上下文+长任务+工具使用"这条 agent 主线的可用性同时提升——内部 Cursor、NVIDIA、MagicPath 等的早期反馈都集中在"在大代码库里持续推进 20 小时级任务的能力"。对自建 coding agent 的团队，意味着可以把更长的工单丢给一次性运行而不是切片喂入。

最可能影响的环节：开发 / 选型 / 测试。

【工程意义】
- 对开发：Codex 内的多文件改造、跨仓库重构、长 PR 合并这类任务可以减少分步骤干预；同时由于"更省 token"，在 Pro 计划内单次能跑更深的任务。
- 对测试：Terminal-Bench 2.0、Expert-SWE 的提升直接对应到"长链路 + 多次工具调用"的回归 eval，建议把内部的长任务回归集再跑一轮做对照。
- 对流程：API 层的 Cyber-permissive 模型仍走 Trusted Access for Cyber 申请，安全工程团队在内部部署前需要把 cyber 类工作流的提示词和审计预案先对齐。

【建议动作】
- 立即可做：在 ChatGPT Codex 上把当前"靠 GPT-5.4 多步推进"的大任务复跑一遍，测一致性与 token 消耗，作为后续切流的基线。
- 本周可验证：等 API 上线后，对你们最痛的一个长上下文 agent 任务（如多文件重构、跨仓库 bug 修复）跑 5.4→5.5 的 A/B，记录"无需人工干预步骤数"和"实际输入/输出 token"，价格涨了但 token 是否真的少。
- 暂时观察：Pro 的 latency 改进与 Fast 模式都是 ChatGPT/Codex 内体验，API 行为可能有差异，关键链路上的切流先在小流量验证。

### 二、Anthropic 公开 Claude Code 三连环 postmortem：模型权重没退化，是 harness、缓存与 system prompt 出问题

1. Anthropic 官方文章 `An update on recent Claude Code quality reports`（URL `engineering/april-23-postmortem`）4 月 23 日发布，配套补救已在 v2.1.116 落地；同时在 X 公告所有订阅用户的用量上限被重置；The Register 4 月 23 日同步报道（标注时间 23 Apr 2026 23:26 UTC，折算北京时间已临近 4 月 24 日凌晨，但 Anthropic 官方公告日期为 4 月 23 日，按官方日期收录）。
2. 三处独立缺陷被拆得很清楚：① 3/4 把 Opus 4.6 / Sonnet 4.6 默认 reasoning effort 从 high 调到 medium，4/7 回退；② 3/26 上线的 thinking-cache 优化把"空闲一小时清一次旧 reasoning"误写成"每一回合都清"，导致 Claude Code 表现得"健忘且重复"，且因 cache miss 让用量上限提前耗尽，4/10 在 v2.1.101 修复；③ 4/16 为配合 Opus 4.7 上线加入 system prompt"工具调用之间 ≤25 词、最终回复 ≤100 词"，事后 ablation 显示对 Opus 4.6/4.7 各掉 3%，4/20 回退。
3. Anthropic 反思了"内部 dogfood + 自动化测试 + code review 都没拦住"，承诺：扩大内部使用公开版本 Claude Code 的比例、对所有 system prompt 改动跑全模型 eval 与 ablation、加 soak period 与灰度、给 model-specific 改动加 CLAUDE.md 守门规则；同时 Claude Code Code Review 工具已能在事后回测中找出该 cache bug，将作为持续防线。

为什么值得工程团队关注：这是少见的"AI 公司公开自家 agent 服务的运维事故"，几乎是一份现成的"运营自家 coding agent 的检查清单"——任何在内部封装 LLM、做 prompt-cache、做 system-prompt 工程、做"多 model 共享 harness"的团队，都能直接拿来对照。

最可能影响的环节：流程 / 测试 / 平台。

【工程意义】
- 对开发：使用 Claude Code / Agent SDK / Cowork 的团队应升级到 v2.1.116+，并把 effort 默认值复核（Opus 4.7 默认 xhigh、其它默认 high）。
- 对测试：内部封装 LLM 的团队，应把"在原始模型权重不变的前提下，仅改 prompt/cache 是否会导致质量回退"加进 eval 套件；这是大多数公司当前 eval 没覆盖的盲区。
- 对流程：把"system prompt 变更 + harness 变更 + cache 策略变更"按"模型权重变更"的同等强度走灰度与 ablation；Anthropic 这次踩的全是这条线。

【建议动作】
- 立即可做：升级 Claude Code 到 v2.1.116+，确认 effort 默认值，并对内部依赖 Claude Code 的脚本回顾一遍是否曾出现"健忘、重复、用量异常"，对应到 3/26-4/10 区间的回归报告。
- 本周可验证：对自家 agent 的"system prompt 改动 + cache 策略改动"加 ablation eval，先复盘最近 30 天内的非模型变更对生产 eval 的影响。
- 暂时观察：The Register 同期还报道 Opus 4.7 因 Acceptable Use Classifier 拒答率上升，是否需要等下一版安全分类器的 patch 再放量。

【来源】
- Anthropic 官方：`An update on recent Claude Code quality reports`（`https://www.anthropic.com/engineering/april-23-postmortem`）
- The Register：`Anthropic admits it dumbed down Claude when trying to make it smarter`（4 月 23 日）

## 国内动态

### 三、DeepSeek 同日开源 DeepEP V2 与 TileKernels：把 MoE 训练侧的通信库和 GPU kernel 一并放出

1. DeepSeek 在 GitHub 上同日发布 `deepseek-ai/DeepEP V2` 与 `deepseek-ai/TileKernels` 两个仓库；时间为北京时间 4 月 23 日，由公开报道与 GitHub 提交记录共同确认；官方未单独发英文博客，因此叙述上以"公开仓库 + 第三方报道"双源核验，单条数字以仓库 README 为准。
2. DeepEP V2 是其 expert-parallel 通信库的整体重写：声称在 router dispatch / combine 阶段相比 V1 取得最高 1.3× 峰值性能、同时 SM 资源占用最高降低 4×；引入 fully JIT 编译、NCCL Gin backend、对 Engram、pipeline parallelism、context parallelism 的支持，以及 EP2048 级别扩展能力；提供"0 SM"系列实验性方案以进一步降低资源开销。
3. TileKernels 是基于 TileLang 写的 GPU kernel 集合，核心覆盖 MoE 路由、gating、量化（FP8/FP4/E5M6 的 per-token / per-block / per-channel）、以及融合 RMSNorm 与梯度的 Engram gating kernel、带 Sinkhorn 归一化的 Manifold HyperConnection kernel；目标是绕开 CUTLASS 等通用库，直接面向 Hopper / Blackwell 的 tile 架构做更高效的矩阵核。

为什么值得工程团队关注：这次开源的不是新模型，而是过去常被关在 NVIDIA、各大云厂内部的"训练系统优化"层。如果团队自己训练或微调 MoE 模型，或者在 H100/H200/B200 上重度做推理优化，这两个仓库提供了可借鉴的实现，同时也降低了"必须用某家闭源训练栈"的门槛。

最可能影响的环节：训练平台 / 推理优化 / 选型。

【工程意义】
- 对开发：自研 MoE 训练框架的团队可以直接评估 DeepEP V2 替换现有 all-to-all 通信路径的可能性，关注的是 SM 占用是否真能降至 4× 以内。
- 对测试：在你们已有的 MoE 训练 / 微调 pipeline 上做 V1→V2 的吞吐与稳定性回归，记录大规模下的实际表现是否符合 1.3× 峰值的报告值。
- 对流程：TileKernels 用 TileLang 而非 CUDA C++，会改变现有"算子开发"的工程门槛——值得评估纳入内部算子库的策略与维护成本。

【建议动作】
- 立即可做：派一名熟悉 MoE 训练的同事 fork DeepEP V2，在小规模 EP 训练上做 V1→V2 的吞吐回归，看 SM 占用与 router 阶段时延是否如官方所述。
- 本周可验证：对 TileKernels 中的量化 kernel（FP8/FP4 per-token/per-block）跑你们最常用的几个 GEMM 形状，对比 cuBLAS / CUTLASS 在相同精度下的延迟与吞吐。
- 暂时观察：Blackwell 系列 kernel 调优本身仍在快速演进，深度集成到生产训练栈前，等社区确认稳定性。

【来源】
- GitHub：`deepseek-ai/DeepEP`（V2 release）
- GitHub：`deepseek-ai/TileKernels`
- 公开报道交叉核验：网易科技 4 月 23 日报道、Startup Fortune 4 月 23 日报道（仅用于核验时间与功能定位，数字以 README 为准）

## 工程工具链

### 四、GitHub 把 cloud agent session 直接挂进 issue / project，并优化 web 端 stack-trace 根因分析

1. GitHub Changelog 4 月 23 日同日发布多条 Copilot 相关条目；本节聚焦其中两条工程价值最高的：`View and manage agent sessions from issues and projects`（Release）与 `Better debugging with GitHub Copilot on the web`（Improvement）；同日还有 `Copilot Chat improvements for pull requests`、`Copilot cloud agent fields added to usage metrics` 等辅助条目。
2. Issues / Projects 视图新增"agent session pill"和侧栏面板：可以从 issue header 直接看到该 issue 上所有 active / completed 的 cloud agent session、点开看进度日志、就地下指令；project 视图默认显示 agent sessions，可在看板上直接点开 session 侧栏；标签选择器也增加了"按使用频率排序"的小优化。
3. 网页端 Copilot Chat 在贴入 stack trace 时给出更结构化的根因分析：固定输出"哪里失败 / 为何失败 / 假设被违反在哪 / 最可能的根因 / 证据代码引用 / 置信度与建议修复 / 后续验证步骤"七段，并明确"贴 stack trace + 仓库/文件上下文 + 复现步骤"为最佳输入。

为什么值得工程团队关注：之前 Copilot cloud agent 在 PR 上的可视化已经比较完整，但回到 issue / project 时仍是脱节的；这次更新意味着"agent 任务的 SDLC 起点"从 PR 进一步前移到了 issue 与 backlog 视图，是把 cloud agent 真正纳入产研流程的关键一步。结构化 stack trace 分析则给一线开发的"日常排错"提供了可重复的操作模式。

最可能影响的环节：流程 / 协作工具链 / 调试。

【工程意义】
- 对开发：日常 stack trace 排错有了固定的"七段式"输出结构，结果更易于贴回 PR 描述或事故复盘文档。
- 对测试：测试团队提的回归 issue 可以直接挂 cloud agent 跑修复尝试，并在 issue 里看 session 日志，缩短"测试报 bug → 工程认领 → 复现 → 修复"的链路。
- 对流程：项目经理 / Tech Lead 在 project 看板上能看到哪些卡片下挂了 agent session，便于盘点"agent 实际承担了多少工单"。

【建议动作】
- 立即可做：在一个内部小项目的 GitHub Project 里把 agent sessions 默认显示打开，挑 2-3 个 backlog issue 试着分配 cloud agent，观察 session 在 issue 与 project 之间的可见性。
- 本周可验证：要求一线工程师把至少 3 次真实排错改用 web Copilot Chat 做 stack-trace 分析，统计七段式输出的命中率（"最可能根因"是否真的对应实际 fix）。
- 暂时观察：cloud agent 在 issue / project 上的可观测性看似无害，但会改变"谁该响应一个 issue"的隐性约定——团队治理流程要同步明确"agent 试过之后人工接管"的边界。
