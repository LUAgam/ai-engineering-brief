## 工程工具链 / 社区动态

### 一、smolagents 提议/实现“工具输出层”间接提示词注入 Shield（可插拔、带阻断/清洗/告警策略）

1. Hugging Face `smolagents` 社区提出并提交 PR：为 `MultiStepAgent`（`CodeAgent`/`ToolCallingAgent` 继承）新增 `shields=` 参数，在**工具输出进入上下文窗口之前**进行扫描，防御“间接提示词注入”（tool output injection）。
2. PR 同时提供两类 Shield：**零依赖的规则检测**（`PatternShield`）与 **基于 Meta Prompt Guard 2 的模型检测**（`PromptGuardShield`，以 optional extra 形式引入），并支持 `BLOCK/SANITIZE/WARN` 三种处理动作。
3. 该变更覆盖 agent 的关键路径（`ToolCallingAgent.process_single_tool_call()`、`CodeAgent._step_stream()`），并附带 57 条测试用例，强调 unicode、长文本、埋藏式注入等边界条件。

2. 为什么值得工程团队关注：很多团队的防护只停留在“执行沙箱/权限最小化”，但**无法阻止模型在被污染上下文驱动下生成危险代码/危险工具调用**；把防线前移到“工具输出进入上下文之前”，能显著降低链路风险，且更容易标准化（框架级能力）。
3. 最可能影响的环节：开发（agent/工具接入方式）、测试（安全回归与误报率评估）、流程（对外部数据源与工具结果的准入与监控）。

【工程意义】
- 对开发：提供统一拦截点，避免每个 tool 各自做输出净化；对 `CodeAgent` 尤其关键（会执行真实 Python）。
- 对测试：可以把“注入样例集”做成固定回归（BLOCK/SANITIZE/WARN 行为可测），并量化误报/漏报。
- 对流程：为“允许联网/允许调用外部 API”的 agent 增加默认防线，便于在平台层做策略下发与审计。

【建议动作】
- 立即可做：梳理现有 agent 工具链中**哪些 tool 输出直接拼进 prompt/上下文**；为高风险工具（浏览器/抓取/检索/SQL/CI 日志）先加一层最小规则过滤与告警。
- 本周可验证：用你们真实的工具输出样本（日志/网页/工单/代码片段）跑一轮“注入样例集”，评估误报；若采用模型检测（如 Prompt Guard 2），评估成本与延迟，并做降级策略（规则预筛→模型复核）。
- 暂时观察：该 PR 仍为未合并状态，需跟踪其 API 设计是否稳定、以及默认策略是否会破坏现有工具输出（特别是代码块/HTML/Markdown 相关场景）。

## 社区动态 / 供应链与协议基础设施（MCP）

### 二、MCP Python SDK：接收原始无效 UTF-8 字节可触发 server 崩溃（DoS 风险），并给出可复现 PoC

1. `modelcontextprotocol/python-sdk` 报告：当 client 在请求参数中发送**原始无效 UTF-8 字节**（例如 `0xFF 0xFE`），MCP Python SDK server 可能在响应路径触发 `anyio.ClosedResourceError` 并崩溃；单次请求即可导致 DoS。
2. 报告给出最小 PoC：通过管道将恶意 JSON-RPC 片段写入 `mcp_server_fetch` 的 stdin，构造含原始字节的 URL 参数，触发崩溃堆栈。
3. 该问题强调了一个工程现实：即使“字符串验证”能处理非法字符，**原始字节层面的输入与传输关闭**也可能走到未覆盖的异常路径，导致服务级中断。

2. 为什么值得工程团队关注：MCP server 往往运行在开发机/CI/内网服务上，具有文件与凭据访问能力；若被恶意 client 或异常网关流量触发崩溃，会造成**工具链不可用**，并可能被用作干扰/掩护其它攻击的手段。
3. 最可能影响的环节：流程（MCP server 稳定性与可观测性）、测试（fuzz/异常输入回归）、开发（对输入与传输错误的统一处理）。

【工程意义】
- 对开发：把 MCP server 当作“面向不可信输入的网络服务”来做防御性编程（输入解码、异常隔离、连接关闭语义）。
- 对测试：应引入“协议层 fuzz + 原始字节输入”测试，而不仅是 JSON 层的字段校验。
- 对流程：需要完善运行时监控（崩溃重启、熔断、速率限制、异常告警），避免一次异常输入拖垮整条 agent 流程。

【建议动作】
- 立即可做：为自建 MCP server 增加**进程级隔离与自动重启**（systemd/容器/监督进程），并在入口加速率限制；对 streamable HTTP/stdio 入口做最小异常捕获与日志。
- 本周可验证：在预发/测试环境复现 PoC 思路（含原始字节输入），验证 server 是否会 crash；同时把“非法 UTF-8/半包/连接中断”加入回归。
- 暂时观察：跟踪 MCP Python SDK 后续修复落地与发布版本；在修复前，避免把 MCP server 暴露在不受控网络边界之外。

## 今日可落地清单
1. 给所有“会把外部数据写入上下文”的 agent 工具调用链加一层统一拦截：先做 `WARN` 级规则扫描与打点，再逐步启用 `SANITIZE/BLOCK`。
2. 把 MCP server 纳入稳定性与安全基线：入口限流 + 崩溃自动重启 + 关键异常告警（ClosedResourceError/解码失败/连接中断）。
3. 建一个“间接提示词注入 + 协议异常输入”的最小回归集（含真实日志/网页片段/乱码字节），每次升级 agent 框架或 MCP SDK 都跑一遍。
