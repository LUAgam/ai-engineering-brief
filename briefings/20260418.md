## 工程工具链

### 一、GitHub Copilot CLI 在北京时间 4 月 18 日早间上线 Auto Model Selection：把模型路由从手工切换变成受策略约束的默认能力

1. GitHub 官方 Changelog《GitHub Copilot CLI now supports Copilot auto model selection》的结构化时间 `datePublished` 为 `2026-04-17T15:32:41-07:00`，折算北京时间为 `2026-04-18 06:32:41`，时间边界成立。  
2. 这次更新把 Copilot CLI 的 `auto` 模式正式开放给全部 Copilot 计划；GitHub 明确说明，CLI 会在 GPT-5.4、GPT-5.3-Codex、Sonnet 4.6、Haiku 4.5 等模型之间，按计划与管理员策略自动挑选“最有效率”的模型。  
3. 官方同时补了三类关键约束：CLI 会直接展示本次实际使用的模型；用户随时可以从 `auto` 切回指定模型；自动选择仍遵守管理员的模型策略；计费上仅在 0x~1x multiplier 范围内选型，且付费用户使用 `auto` 时享有 10% multiplier 折扣。

为什么值得工程团队关注：很多团队已经开始把 AI CLI 接入日常开发，但最大的不稳定点不是“有没有模型”，而是不同任务在速度、成本、可用性之间的切换策略仍靠工程师手动决定。GitHub 这次把模型路由、策略继承和计费规则显式做进 CLI，本质上是在把“模型选型”从个人习惯收敛成平台能力。

最可能影响的环节：开发 / 测试 / 流程 / 成本治理 / CLI 选型策略。

【工程意义】
- 对开发：日常问答、脚本生成、代码修改和命令解释不必每次手工挑模型，减少上下文切换和误选高成本模型的概率。
- 对测试：可以把“同一任务在 auto 与固定模型下的成功率、耗时、请求消耗”纳入回归比较，而不是只看主观体验。
- 对流程：管理员策略仍然生效，说明自动路由可以更容易进入企业默认配置，而不是沦为个人开发者特性。

【建议动作】
- 立即可做：在团队里选 3~5 个高频 CLI 任务，记录当前固定模型用法、平均耗时和 premium request 消耗，作为后续对照基线。
- 本周可验证：升级 Copilot CLI 后，对比 `auto`、固定 GPT-5.3-Codex、固定 Sonnet 4.6 在真实仓库中的任务成功率、响应延迟与成本差异。
- 暂时观察：先不要把 `auto` 直接作为所有高风险改动的默认模式，特别是需要稳定复现输出的脚手架生成、批量重构和 CI 辅助场景。

【来源】
- GitHub Changelog：`GitHub Copilot CLI now supports Copilot auto model selection`

## 生成式AI

### 二、Google 在北京时间 4 月 18 日凌晨发布 Android 混合推理方案：同一套 API 开始支持端侧 Gemini Nano 与云端 Gemini 动态切换

1. Android Developers Blog 官方 RSS 中，《Experimental hybrid inference and new Gemini models for Android》的 `pubDate` 为 `Fri, 17 Apr 2026 20:00:00 +0000`，折算北京时间为 `2026-04-18 04:00:00`，时间边界成立。  
2. Google 在 Android 侧发布了 Firebase AI Logic 的 hybrid inference API，允许应用通过统一接口在设备本地的 Gemini Nano 与云端 Gemini 模型之间切换；官方示例给出 `PREFER_ON_DEVICE` 和 `PREFER_IN_CLOUD` 两种模式，并明确依赖版本为 `com.google.firebase:firebase-ai:17.11.0` 与 `firebase-ai-ondevice:16.0.0-beta01`。  
3. 同一篇更新还把新模型接入 Android 场景：包括 Gemini 3.1 Flash-Lite，以及 Nano Banana 2 图像生成/编辑能力；Google 明确展示了用新模型改造 Magic Selfie 示例，把背景分割交给生成模型本身处理，以简化实现路径。

为什么值得工程团队关注：移动端 agent 和 AI 功能落地，长期卡在“什么时候必须上云、什么时候可在端上兜底、离线和低网环境怎么处理”这些工程问题。Google 这次不是单纯发模型，而是把端云切换、回退策略和 SDK 依赖都做成了可用的开发入口，对真正要把 AI 能力塞进 Android 应用的团队更有现实意义。

最可能影响的环节：开发 / 测试 / 流程 / 移动端架构 / 成本与延迟治理。

【工程意义】
- 对开发：端侧与云侧能力不再需要两套调用栈维护，移动团队可以先围绕统一 API 做功能原型，再逐步细化路由策略。
- 对测试：可以正式建立“在线/离线、支持 Nano/不支持 Nano、弱网/强网”矩阵，验证回退路径是否稳定，而不是只测云端 happy path。
- 对流程：如果团队已经使用 Firebase AI Logic，这次更新意味着模型接入、设备能力判断和演进路径更容易纳入现有 Android 发布流程。

【建议动作】
- 立即可做：梳理当前 Android AI 功能里哪些请求必须上云，哪些任务可以接受端侧降级输出，并补出网络/机型分层表。
- 本周可验证：用一个真实功能试跑 hybrid inference，分别测 `PREFER_ON_DEVICE` 与 `PREFER_IN_CLOUD` 在成功率、首字延迟、耗电与离线可用性上的差异。
- 暂时观察：官方已明确该 API 仍属 experimental，先不要把高价值业务逻辑完全绑定在单一 hybrid routing 策略上，重点观察设备兼容性与多轮任务能力边界。

【来源】
- Android Developers Blog：`Experimental hybrid inference and new Gemini models for Android`
- Android Developers RSS feed（用于确认精确发布时间）

### 三、Google Developers Blog 发布 A2UI v0.9：生成式 UI 开始从“演示效果”转向可接入现有设计系统的协议层

1. Google Developers Blog《A2UI v0.9: The New Standard for Portable, Framework-Agnostic Generative UI》页面仅公开 `datePublished: 2026-04-17`，未公开精确时刻；按编辑部规则，Google 属于美西时区厂商，因此**发布时间可能为北京时间 2026-04-18**。该条按“官方日期成立、精确时刻未公开”的保守口径收录。  
2. A2UI v0.9 的核心不是再造一套前端框架，而是把 agent 输出的 UI intent 与具体框架解耦。官方这次新增 shared `web-core`、React 官方 renderer、A2UI Agent SDK，并把可选组件集从“Standard”改名为更接近现有设计系统语义的“Basic”。  
3. 工程层面最值得看的是协议与运行时能力：Google 明确写到支持 `pip install a2ui-agent-sdk` 快速接入；支持 version negotiation、dynamic catalogs、resilient streaming；并把 transport 简化到可走 MCP、WebSockets、REST、AG-UI、A2A 等多种通道。

为什么值得工程团队关注：很多团队现在做“AI 生成前端”仍然停留在让模型直接吐 React/Vue 代码，结果是组件漂移、设计系统失控、渲染协议不统一。A2UI v0.9 更像是在补一层 agent 与前端之间的稳定契约：模型表达意图，客户端决定如何在既有组件体系中渲染，这对需要维护设计系统、权限边界和跨端一致性的团队更重要。

最可能影响的环节：开发 / 测试 / 流程 / 前端协议层 / 设计系统接入。

【工程意义】
- 对开发：如果团队已有组件库和 catalog，A2UI 这类协议比“让模型直接拼 JSX”更容易控制渲染边界和升级路径。
- 对测试：resilient streaming、schema version negotiation 与 dynamic catalogs 更适合做契约测试、组件回放和跨端一致性检查。
- 对流程：A2UI over MCP / REST / WebSockets 说明生成式 UI 可以被纳入现有网关、权限、审计和前后端协作流程，而不必单独养一套旁路系统。

【建议动作】
- 立即可做：盘点团队现有 design system 是否已有结构化 catalog、属性约束和跨端映射；如果没有，先补这一层，再谈 agent 直接驱动 UI。
- 本周可验证：选一个表单或配置面板场景，试做“catalog + A2UI schema + renderer + streaming 回放”的最小闭环，而不是直接让模型生成页面代码。
- 暂时观察：由于官方页面未公开精确发布时间，且 A2UI 仍在快速演进，本周更适合作为协议层调研对象，不建议立即作为生产主路径强绑定。

【来源】
- Google Developers Blog：`A2UI v0.9: The New Standard for Portable, Framework-Agnostic Generative UI`
- Google Developers RSS / 页面结构化数据（用于确认官方日期）
