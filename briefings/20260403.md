### 一、Cursor 3 在北京时间 4 月 3 日进入正式发布口径，重点补齐多仓并行、隔离工作树与团队级治理

1. Cursor 官方博客《Meet the new Cursor》和官方 Changelog《New Cursor Interface》页面日期均为 `Apr 2, 2026`；页面未给出更精确时区，按标准版规则对美西厂商采用默认总部时区保守判断，落入北京时间 `2026-04-03` 的可能性成立，但时间边界清晰度低于带 UTC 时间戳的 GitHub Release。  
2. 这次发布不是界面换皮，而是把 Cursor 重新组织成以 agent 为中心的统一工作台：新增 **Agents Window**，支持跨 repo、worktree、本地、云端和 remote SSH 并行运行 agent；新增 **Design Mode** 直接在浏览器中圈选 UI 元素给 agent；并把本地与云端 agent 的会话交接做成显式能力。  
3. Changelog 里真正对工程团队更关键的是运行与治理细节：新增 `/worktree` 与 `/best-of-n` 命令做隔离试验与并行比较；企业侧增加 team secret 仅管理员可改、审计日志可读性提升、可关闭 “Made with Cursor” attribution；运行时新增 `Await` 工具等待后台命令/子代理完成，并收紧 browser automation tool surface 与 subagent 的浏览器工具使用范围。

2. 为什么值得工程团队关注：这说明 Cursor 的重点已经明显从“单个会话里帮你补代码”转向“让多个 agent 在真实团队流程里可并行、可隔离、可交接、可审计”。对正在评估多 agent 工作台的团队，`/worktree`、`/best-of-n`、Await、团队 secret 治理这些能力，比单次生成质量更接近日常落地门槛。

3. 最可能影响的环节：开发 / 测试 / 流程 / 安全治理 / 选型。

【工程意义】
- 对开发：多仓、多 worktree、本地与云端之间的 handoff 更顺，适合把长任务、并行探索和临时修复拆开处理。
- 对测试：应把 `/worktree`、`/best-of-n`、Await、Design Mode、browser fallback 和大文件 diff 性能纳入 focused regression，而不是只测单轮代码生成。
- 对流程：团队级 secret 权限、审计日志可读性、归因控制等变化，说明 Cursor 已在补企业接入与合规治理短板。

### 二、Claude Code v2.1.91 在北京时间 4 月 3 日发布，继续把“批准策略、恢复链路、技能执行”往可控方向收紧

1. `anthropics/claude-code` 官方 GitHub Release `v2.1.91` 的 `published_at` 为 `2026-04-02T23:45:35Z`，折算北京时间为 `2026-04-03 07:45:35`，时间边界成立。  
2. 这版最关键的新能力不是新命令，而是把权限与上下文边界做细：新增 MCP 工具结果持久化覆盖 `_meta["anthropic/maxResultSizeChars"]`，允许更大的结果（上限 500K）不被截断；新增 `disableSkillShellExecution`，可禁用 skills、自定义 slash command 和插件命令里的内联 shell 执行；插件也开始支持在 `bin/` 下自带可执行文件并以裸命令调用。  
3. 同版还修了几类真实流程里容易踩坑的点：`--resume` 在异步 transcript 写入失败时可能断链并丢历史；远程会话里的 plan mode 在容器重启后会丢 plan file 跟踪，进而触发多余权限弹窗和空白审批界面；`permissions.defaultMode: "auto"` 的 JSON schema 校验也被修正。

2. 为什么值得工程团队关注：Claude Code 近几版的重点很明显，不是继续加入口功能，而是在补“允许什么”“恢复后还记不记得上下文”“skill 是否偷偷带 shell”“远程 plan mode 会不会失控”这些真正决定团队能否放进日常流程的问题。尤其是 `disableSkillShellExecution`，直接对应企业对 skill/plugin 扩展面的最小权限要求。

3. 最可能影响的环节：开发 / 测试 / 流程 / 安全治理。

【工程意义】
- 对开发：大结果不再轻易被截断，数据库 schema、长诊断信息或大型工具输出更容易完整进入后续推理链。
- 对测试：应重点补 `--resume` transcript 连续性、远程 plan mode 容器重启、`disableSkillShellExecution`、插件 `bin/` 执行与权限策略的组合回归。
- 对流程：如果团队在推广 skill/plugin 生态，这版给出的信号是必须把“扩展面是否能执行 shell”纳入治理基线。

【建议动作】
- 立即可做：审查现有 Claude Code skill、插件和 slash command，确认是否需要默认关闭 inline shell execution。
- 本周可验证：在远程容器/开发环境中模拟 plan mode + 重启 + resume，验证 plan file 跟踪、审批提示和 transcript 是否连续。
- 暂时观察：继续观察后续版本是否继续围绕 transcript、plan mode、plugin 权限和大型工具结果处理做回归修复。

### 三、Gemini CLI v0.37.0-preview.1 在北京时间 4 月 3 日发布预览修补版，把“持久批准”改成按模式传播的上下文感知策略

1. `google-gemini/gemini-cli` 官方 GitHub Release `v0.37.0-preview.1` 的 `published_at` 为 `2026-04-02T21:09:35Z`，折算北京时间为 `2026-04-03 05:09:35`，时间边界成立；但该版本为 **pre-release**，结论应按预发布口径理解。  
2. 单页 release note 只有一个 patch 说明，但追到官方 compare 与对应 PR 后可以确认，这个版本的核心改动是 **context-aware persistent policy approvals**：把持久批准与当前 approval mode 绑定，形成 `plan < default < autoEdit < yolo` 的 permissiveness hierarchy。也就是说，在更受限模式里授予的信任不会被错误地下沉到更敏感场景，而在 plan mode 做出的全局信任选择才会显式扩散到所有模式。  
3. 这次 patch 同时补了策略持久化与调度层的实现和测试：规则现在可写入 `modes`，已有规则会被更新而不是重复追加；文档明确了 plan mode 作为“安全研究环境”时的边界；相关 persistence / scheduler tests 也一并补上。

2. 为什么值得工程团队关注：这类 patch 看起来小，但它直指 agent 工具最难处理的一类问题——一次“总是允许”到底在什么上下文里生效。如果批准策略不区分 plan、default、autoEdit、yolo，就很容易把原本只想在某个受控模式内放行的工具，错误扩散到更高风险执行路径。

3. 最可能影响的环节：开发 / 测试 / 流程 / 安全治理 / 选型。

【工程意义】
- 对开发：在 plan mode 做研究、在 default/autoEdit 做落地时，工具批准边界更符合直觉，减少“研究阶段放过一次，执行阶段就全局信任”的误伤。
- 对测试：应重点覆盖 approval mode 迁移、持久策略写回、已有规则更新、交互式批准升级与回放一致性。
- 对流程：如果团队在评估 CLI agent 的审批系统，不应只看是否有 allow/deny，还要验证批准是否具备上下文作用域。

【建议动作】
- 立即可做：把 “same tool, different approval modes” 纳入 Gemini CLI 的最小验收集，验证 plan/default/autoEdit/yolo 的授权传播是否符合预期。
- 本周可验证：检查你们内部 agent 工具是否也存在“允许一次即全局生效”的策略漏洞，并参考 Gemini CLI 的 mode hierarchy 设计做收敛。
- 暂时观察：由于是 preview 版，先观察该策略是否原样进入后续稳定版，以及是否继续补审批 UI/审计可见性。

【来源】
- GitHub Releases：`google-gemini/gemini-cli v0.37.0-preview.1`
- GitHub PR / Compare：`#24561`、`v0.37.0-preview.0...v0.37.0-preview.1`

### 四、微软在北京时间 4 月 3 日公开 Agent Governance Toolkit，把 runtime security、SRE 与合规校验做成可插现有框架的开源治理层

1. Microsoft Open Source Blog 文章《Introducing the Agent Governance Toolkit: Open-source runtime security for AI agents》页面日期为 `April 2, 2026`；页面未给出更精确时间，按标准版规则对美西厂商保守折算，可视为落入北京时间 `2026-04-03`，但时间边界清晰度低于带 UTC 时间戳的发布源。  
2. 官方把它定义为微软组织下、MIT 协议的开源项目，目标是给自治 agent 提供 runtime governance，并声称以确定性、`<0.1ms p99` 的策略执行覆盖 OWASP 2026 Agentic AI Top 10 的全部 10 类风险。项目拆成 7 个包：Agent OS、Mesh、Runtime、SRE、Compliance、Marketplace、Lightning，分别覆盖策略引擎、身份与信任、执行环、SLO/熔断、合规证据、插件供应链和 RL 训练治理。  
3. 更重要的是它不是另起炉灶重造 agent 框架，而是强调可插入 LangChain、CrewAI、Google ADK、Microsoft Agent Framework、Azure AI Foundry Agent Service、OpenAI Agents SDK、LangGraph、Haystack、PydanticAI 等现有生态，并同时提供 Python、TypeScript、Rust、Go、.NET 多语言包。

2. 为什么值得工程团队关注：这是少见把 agent 治理问题直接工程化为“策略执行 + 身份信任 + SRE + 合规证据 + 插件签名”的官方开源方案。对平台团队和安全团队来说，它提供了一个更接近生产环境的参考实现，而不是停留在“最佳实践建议”层面。

3. 最可能影响的环节：开发 / 测试 / 流程 / 安全治理 / 合规。

【工程意义】
- 对开发：如果你们已经在用 LangGraph、OpenAI Agents SDK、Foundry Agent Service 等框架，可以把治理层当作附加能力评估，而不必推倒重来。
- 对测试：值得补 agent action interception、kill switch、trust scoring、plugin signing、policy regression、chaos engineering 和 SLO enforcement 相关验证。
- 对流程：它把 agent 从“功能接得上”推进到“谁能执行、何时中止、如何审计、如何留证”这些企业流程问题。

【建议动作】
- 立即可做：把你们现有 agent 系统按“策略引擎 / 身份信任 / 熔断 / 审批 / 合规留证 / 插件签名”六个维度做一次缺口盘点。
- 本周可验证：选一个现有 agent workflow，评估是否能以 sidecar / middleware 方式接入类似治理层，而不是把控制逻辑散落在业务代码里。
- 暂时观察：继续关注该项目是否形成稳定社区与上游集成，尤其是对 OpenAI Agents SDK、LangGraph、Foundry 的实际维护深度。

【来源】
- Microsoft Open Source Blog：`Introducing the Agent Governance Toolkit: Open-source runtime security for AI agents`

### 五、OpenAI 在北京时间 4 月 3 日宣布 Codex 团队按量计费，编码 agent 试点从“席位前置”变成“按工作流归因”

1. OpenAI 官方公告《Codex now offers pay-as-you-go pricing for teams》页面日期为 `April 2, 2026`；正文未给出更精确时区/发布时间，按标准版规则对美西厂商作保守判断，可视为落入北京时间 `2026-04-03`，但时间边界清晰度低于带 UTC 时间戳的 GitHub Release。  
2. 这次变化的核心不是模型能力，而是团队接入方式：ChatGPT Business / Enterprise 现在可添加 **Codex-only seats**，无固定 seat fee、按 token 用量计费、且无 rate limits；同时 ChatGPT Business 年价从 `25` 美元/席位降到 `20` 美元/席位。  
3. OpenAI 还把接入路径说得更明确：推荐通过 Codex app for macOS / Windows 使用，并把 Plugins 与 Automations 作为连接团队现有系统的主路径；对合格的 Business 工作区，还提供每个新增 Codex-only 成员 `100` 美元 credits、每队最高 `500` 美元的限时试点激励。

2. 为什么值得工程团队关注：很多编码 agent 的真正阻力不是“好不好用”，而是“怎么小范围试、怎么按工作流归因、怎么跟现有预算模型对接”。OpenAI 这次改动把 Codex 的试点门槛从固定席位采购转成按量试验，更适合平台团队先围绕具体工作流验证 ROI。

3. 最可能影响的环节：开发 / 流程 / 成本治理 / 选型。

【工程意义】
- 对开发：更适合先围绕高频编码、修复、脚本生成、PR 辅助等具体工作流做小范围接入，而不是一次性全员开通。
- 对测试：应把 token 消耗、任务成功率、失败重试率、插件/自动化链路稳定性一起纳入试点评估。
- 对流程：平台团队可以更容易按团队、项目或工作流拆账，建立预算上限与灰度准入，而不是把成本埋在统一席位里。

【建议动作】

- 暂时观察：继续观察 OpenAI 是否补上更细的团队治理、权限控制、审计与自动化运维能力；若只有定价变化，扩面仍要谨慎。
