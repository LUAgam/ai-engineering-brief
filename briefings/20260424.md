## 生成式AI

### 一、DeepSeek 开源 V4 系列：1M 上下文 + 混合注意力 + FP4 QAT，1.6T Pro 与 284B Flash 一并放出

1. DeepSeek 4 月 24 日在 Hugging Face 同时发布 DeepSeek-V4-Pro（1.6T 总参 / 49B 激活）与 DeepSeek-V4-Flash（284B 总参 / 13B 激活）两款 MoE 模型，权重 MIT 授权，含 V4-Pro / V4-Pro-Base / V4-Flash / V4-Flash-Base 四个 checkpoint；NVIDIA Developer Blog 同日发布 Day-0 部署指南，DeepSeek 官方 API 同步上线，1M 原生上下文、最大 384K 输出。
2. 关键架构变化在长上下文：混合注意力 = Compressed Sparse Attention（CSA，含 Lightning Indexer + DSA）与 Heavily Compressed Attention（HCA）交错；论文给出的口径是 1M 上下文下 V4-Pro 单 token 推理 FLOPs 仅为 V3.2 的 27%、KV 缓存仅 10%，V4-Flash 分别为 10% 与 7%。残差被替换为 Manifold-Constrained Hyper-Connections（用 Sinkhorn-Knopp 约束在 Birkhoff 多面体上），主参数改用 Muon 优化器，MoE 专家与 CSA 的 QK 路径做 MXFP4 QAT，推理时直接用真 FP4 权重。
3. 性能与生态对位：V4-Pro-Max 在 Codeforces 3206、SWE-Verified 80.6%、OpenAI MRCR 1M 83.5 MMR；NVIDIA 给出 GB200 NVL72 上 V4-Pro 单用户 > 150 tokens/sec 的初步实测，vLLM、SGLang 均提供 Blackwell / Hopper 上的低延迟、平衡、最大吞吐三类配方与多机 prefill/decode 解耦方案，NIM 也 Day-0 接入。

为什么值得工程团队关注：这是 2026 年至今最硬的"开放权重 + 长上下文 + agent 友好"组合。对自建或自托管 LLM 的团队，意味着 1M 上下文不再必然绑死在闭源 API 上；对 RAG、多文件代码理解、长跨度 agent 这些核心负载，KV 10× 压缩会直接落到推理成本与并发承载上。

最可能影响的环节：选型 / 推理平台 / 长上下文 RAG / 编码 agent。

【工程意义】
- 对开发：需要长上下文的代码 agent、文档 agent、Deep Research 类工作流，多了一个 MIT 可商用、且与 vLLM/SGLang/NIM 对齐的选项，可作为 Claude/Gemini 长上下文的开源对照。
- 对测试：建议将 V4-Pro/Flash 接入现有"长任务回归套件"（多步 tool calling、跨文件代码理解、长检索综合），重点观察 1M 上下文下的事实一致性与 KV 占用是否符合官方 27% / 10% 的曲线。
- 对流程：FP4 QAT + Muon + mHC 的训练侧打法会影响内部模型训练或微调的硬件预算与算子选型，值得训练平台同事提前评估。

【建议动作】
- 立即可做：在 build.nvidia.com 上跑 V4-Pro，挑 2-3 个真实长上下文用例（50 万 token 文档问答、近 1M 全仓库代码提问）与现有 Claude Opus 4.6/4.7、Gemini 3.1 Pro 横评。
- 本周可验证：在自家 Blackwell / Hopper 节点上用 vLLM 或 SGLang recipe 起 V4-Flash，做一次 tokens/sec/$ 核算，评估是否可承载内部 agent 任务。
- 暂时观察：mHC、Muon、FP4 QAT 还需在社区独立复现；自研训练栈大规模采用前先等 1-2 个第三方复现案例。

### 三、Cursor 3.2 发布：Multitask 子智能体、Worktrees、跨仓 Multi-root Workspaces 一并进 Agents Window

1. Cursor 3.2 changelog `Multitask, Worktrees, and Multi-root Workspaces` 4 月 24 日发布，三项变更全部围绕 Agents Window：`/multitask` 把请求展开为多个异步 subagent 并行执行；worktrees 在 Agents Window 内得到完整集成；workspace 现在支持多个根目录作为单个 agent session 的目标。Cursor 总部位于美西（PT），按时区折算实际可能落在北京时间 4 月 24 日下午到 4 月 25 日凌晨，但官方 changelog 日期为 4 月 24 日，按官方日期收录。
2. Multitask 不只是把队列里的请求并行化，还会"把大任务自动拆成更小块，由一组异步 subagent 同时推进"；如果队列里已有未处理消息，可以让 Cursor 直接对队列里的消息开 multitask，而不必等当前 run 完成。
3. Worktrees 支持把不同分支上的隔离任务放到后台执行，需要本地验证时一键把分支拉到前台；Multi-root Workspaces 让单个 agent session 可以横跨前端、后端、共享库等多个仓库做改动，避免"agent 进出仓时要重新 retarget"的来回成本。

为什么值得工程团队关注：这一组变更把 Cursor 3 的"agent-first IDE"叙事真正落到日常多人协作的场景——典型痛点 monorepo 拆仓、并行修 bug、长 worktree 隔离测试，以前都得团队自己用脚本拼接，现在被直接装进 IDE。

最可能影响的环节：开发 / 协作 / 流程。

【工程意义】
- 对开发：多仓重构、跨服务前后端联调可以在一个 agent session 完成，不必再为每个仓库新开窗口。
- 对测试：worktrees 的"一键拉前台"让"agent 跑完背景任务、人工接手验证"的回路缩短，可以把"每个 PR 在隔离 worktree 跑一组冒烟"做成默认流程。
- 对流程：multitask 让一名工程师同时占用多个 agent 配额，团队需要明确"哪些任务允许 multitask、哪些必须串行"，避免共享 Pro 配额被打满。

【建议动作】
- 立即可做：升级到 Cursor 3.2，在一个跨前后端的真实需求上试 `/multitask`，记录子任务拆分粒度是否符合预期；并把至少一个跨仓改动放到 multi-root workspace 里跑。
- 本周可验证：在团队 CI / 本地测试链路里把 worktrees 接管"长跑测试"agent，验证是否能在不阻塞主分支的情况下做隔离回归。
- 暂时观察：multitask 在大型仓库下对 token 消耗与 rate limit 的实际放大倍数，先在小范围观测再放开权限。

【来源】
- Cursor Changelog 3.2：`Multitask, Worktrees, and Multi-root Workspaces`（`https://cursor.com/changelog/04-24-26`，2026-04-24）
- Cursor 官方文档：`worktrees`、`Agents Window`

### 四、GitHub Copilot 个人交互数据训练策略今日生效；同日预告 GitHub App installation token 格式变化

1. GitHub 3 月 25 日预告的政策今日（4 月 24 日）正式生效：Copilot Free / Pro / Pro+ 用户的交互数据（输入、输出、采纳或修改、光标上下文、文件名、仓库结构、与 Copilot 功能交互、反馈打分）默认用于模型训练；Business / Enterprise / 学生计划不在范围。个人用户可在 `github.com/settings/copilot` 的 Privacy 区关闭。私有仓库内容"at rest"不被使用，但 Copilot 实际运行时处理过的私有仓库代码片段会被纳入。
2. GitHub Changelog 4 月 24 日另发 `Notice about upcoming new format for GitHub App installation tokens`：自 4 月 27 日起分阶段把 GitHub App installation token（`ghs_` 前缀）从固定 40 字符改为 `ghs_APPID_JWT` 形式，整体长度约 520 字符；先对 GitHub Actions `GITHUB_TOKEN` 与一方集成（Dependabot、Slack、Teams 等）放量，5 月中—6 月底再向所有 App 推开；GitHub Enterprise Server 不受影响。官方明确不要硬编码 40 字符或 `ghs_[A-Za-z0-9]{36}` 正则。
3. 同日还发布 `Changes to notification retention and archived repository watches`：网页通知保留期从 5 个月缩短到 3 个月（邮件通知不变）；非协作者对归档超过 6 个月的仓库的 watch 将被移除，重新解档后可一键重订阅。

为什么值得工程团队关注：这是同一天的"治理 + 集成层"双重信号。一边是个人用户写公司代码的真实合规风险——Copilot 个人账号是否在内部仍允许使用，需要今天就有结论；一边是对所有自建 GitHub App、CI workflow 与 token 校验逻辑的硬兼容性事件——任何写死 40 字符长度或旧正则的代码都得在 5 月之前完成迁移。

最可能影响的环节：合规 / 平台 / CI。

【工程意义】
- 对开发：使用个人 Copilot 账号写工作仓库代码的同事需要立即检查 Privacy 设置；团队可考虑明确"工作账号必须用 Business/Enterprise 计划"。
- 对测试：在 5 月中之前对所有依赖 GitHub App installation token 的工具链（CI 自动化、内部 bot、Action）做兼容性测试，校验列宽 ≥ 520、不要硬校验 token 长度或前缀长度。
- 对流程：把"通知 3 个月即清"和"归档仓库 watch 自动清理"纳入团队对 GitHub 通知的运营预期，避免下游报表和复盘依赖更长保留窗口。

【建议动作】
- 立即可做：发一条全员通知，要求使用 Copilot 个人计划的人今天就到 `github.com/settings/copilot` 完成 training opt-out，并明确公司代码不应用个人 Copilot 写。
- 本周可验证：扫描内部代码库与基础设施配置是否存在 token 长度断言或 `ghs_` 旧正则；在 4 月 27 日起的灰度里完成修复，并用一份新格式 token 做端到端联调。
- 暂时观察：GitHub 后续会发关于 user-to-server token（Copilot code review 流程）的格式变化指引，相关链路先标记跟踪即可。
