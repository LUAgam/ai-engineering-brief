### 一、OpenAI 以美西 4 月 10 日公告回应 Axios 供应链事件：轮换 macOS 签名证书，并要求 ChatGPT Desktop / Codex / Atlas 升级

1. OpenAI 官方公告页仅显示 `April 10, 2026`；按 OpenAI 主站常见美西日期口径保守换算，可对应北京时间 `2026-04-11`，自然日边界可成立，但具体时分未披露。  
2. OpenAI 披露，`2026-03-31 UTC` Axios 恶意版本曾在其 macOS app-signing GitHub Actions 工作流中被下载执行；虽然当前未发现用户数据、产品或知识产权被攻破证据，但仍将旧证书按“可能暴露”处理，并执行吊销与轮换。  
3. 公告给出明确升级门槛与截止时间：自 `2026-05-08` 起，旧签名版本将不再获得支持且可能无法运行；最早安全版本分别为 `ChatGPT Desktop 1.2026.051`、`Codex App 26.406.40811`、`Codex CLI 0.119.0`、`Atlas 1.2026.84.2`。OpenAI 同时明确根因包括 GitHub Actions 使用 floating tag，且未设置 `minimumReleaseAge`。

为什么值得工程团队关注：这不是“某家 AI 厂商发了安全声明”这么简单，而是把桌面 AI 客户端、CLI agent 和发布流水线串成了一个完整的供应链问题。对自己也在发桌面应用、CLI 工具、插件或 agent runtime 的团队来说，签名与 notarization 工作流已经不是发布末梢，而是产品安全的一部分。

最可能影响的环节：开发 / 流程 / 安全治理 / 桌面分发 / CLI 发布。

【工程意义】
- 对开发：如果你们维护桌面端或 CLI agent，证书轮换、版本门槛和升级指引需要被当成一等发布能力，而不是出事后手工处理。
- 对测试：应补一组发布链路回归：恶意依赖进入 CI、签名证书替换、旧签名阻断、升级提示、自动更新与回滚。
- 对流程：floating tag、`latest` 依赖和过新的包立即可用，已经不适合带签名物料或发布权限的流水线。

【建议动作】
- 立即可做：审计所有签名、notarization、打包和发布工作流，找出仍在使用 floating tag、未锁 commit、未设包龄门槛的步骤。
- 本周可验证：演练一次“恶意 npm 依赖 + 证书轮换 + 旧版本失效”的桌面/CLI 发布事故响应。
- 暂时观察：关注 OpenAI 后续是否公开更细的发布链路 hardening 基线，例如 attestation、SBOM、隔离签名机与更严格的 artifact provenance。

### 二、Microsoft 在北京时间 4 月 11 日发布 Agent Framework .NET 1.1.0：把技能发现、资源脚本类型和 Foundry 会话管理继续往正式工程化推进

1. `microsoft/agent-framework` 官方 GitHub Release `dotnet-1.1.0` 的 `published` 时间为 `2026-04-10T17:17:24Z`，折算北京时间为 `2026-04-11 01:17:24`，时间边界成立。  
2. 这版最关键的新增是把 class-based skills 的资源与脚本发现改成支持 reflection discovery，并补上 skill resource / script functions 的 custom types 支持；这意味着 .NET 团队在构建技能系统时，可以减少手工注册和类型胶水。  
3. 同版还补了几处更贴近落地的问题：`FoundryAgent` 新增 `CreateSessionAsync(conversationId)`，Executor 增加 message delivery callback overloads，修复 checkpoint restoration 输入信号、chat history compaction duplication、Concurrent Workflow sample 和 Computer Use sample 等问题，并把样例模型更新到 `gpt-5.4-mini`。

为什么值得工程团队关注：很多多 Agent 或技能系统到真正接入业务时，问题不在“能不能调模型”，而在技能发现是否稳定、会话 ID 能否和业务对话串起来、checkpoint / compaction 是否会在真实 workflow 里出错。`.NET 1.1.0` 继续把这些基础能力往“少写胶水、少踩边角 bug”的方向收口。

最可能影响的环节：开发 / 测试 / 流程 / 平台选型 / .NET Agent 基础设施。

【工程意义】
- 对开发：class-based skills 的反射发现与 custom types 支持，能减少技能注册模板代码和资源脚本参数转换代码。
- 对测试：应把 checkpoint 恢复、compaction 后历史一致性、Foundry 会话复用、Concurrent Workflow sample 和 Computer Use sample 补进回归。
- 对流程：如果内部平台想统一技能规范和会话治理，这类 release 比单纯换模型更值得优先跟进。

【建议动作】
- 立即可做：拿一个现有 .NET skill/agent 项目验证 reflection discovery 与 custom types，确认是否能删掉现有手工注册代码。
- 本周可验证：补测 `CreateSessionAsync(conversationId)`、checkpoint restore、历史压缩后重复消息、长 workflow callback 交付。
- 暂时观察：继续观察 Microsoft 后续是否把 .NET / Python 两侧的技能、checkpoint 和 Foundry 约束进一步对齐成跨语言默认契约。

【来源】
- GitHub Releases：`microsoft/agent-framework dotnet-1.1.0`

### 三、Google 在北京时间 4 月 11 日发布 Gemini CLI nightly 预发布：集中修补长输出 OOM、PTY 耗尽与 MCP 子进程泄漏

1. `google-gemini/gemini-cli` 官方 GitHub Release `v0.39.0-nightly.20260411.0957f7d3e` 的 `published` 时间为 `2026-04-11T02:08:00Z`，折算北京时间为 `2026-04-11 10:08:00`，时间边界成立。该版本为 nightly 预发布，不应等同于稳定版。  
2. 这版最重要的信号不是新功能，而是一组稳定性修补：修复 lifecycle memory leaks、移除可能导致大输出流 OOM 的 buffer slice、修复 PTY exhaustion 与 orphan MCP subprocess leaks，并把 `/settings` 标记为不安全并发命令。  
3. 同版还补了几项对真实 CLI rollout 很关键的细节：更新时保留 shell execution config、交互 shell 执行传递 session id、持久化 subagent `agentId` 到 tool call records、新增 large memory regression test，并用 native ACL 优化 Windows sandbox 初始化。

为什么值得工程团队关注：这说明 terminal agent 进入团队工作流之后，最容易拖垮体验的已经不是模型回答质量，而是长输出、长会话、子进程清理、并发配置修改和跨平台沙箱启动。这些问题如果不被提前 soak test，很容易在真实仓库或 CI 里累积成“偶发挂死”。

最可能影响的环节：开发 / 测试 / 流程 / CLI 运维 / Windows 环境。

【工程意义】
- 对开发：session id 与 subagent `agentId` 持久化更有利于把 CLI 日志、tool call 和上层审计记录串起来。
- 对测试：应单列覆盖大输出流、长时间 PTY 占用、MCP 子进程回收、`/settings` 并发调用、Windows sandbox 初始化时延。
- 对流程：因为这是 nightly 预发布，更适合作为 canary 和回归样本，而不是直接全量升级基线。

【建议动作】
- 立即可做：给内部 CLI agent 增加一组 soak test，覆盖大输出、MCP 子进程泄漏、PTY 数量上限和并发配置修改。
- 本周可验证：在 Windows runner 上跑一次长会话与沙箱初始化压测，记录内存、PTY、子进程与会话恢复指标。
- 暂时观察：等待后续稳定版确认这些修复是否完整回收，再决定是否纳入团队默认版本。

【来源】
- GitHub Releases：`google-gemini/gemini-cli v0.39.0-nightly.20260411.0957f7d3e`

### 四、OpenClaw 在北京时间 4 月 11 日发布 v2026.4.10：把 Codex provider、Active Memory、exec policy 和 live QA lane 一次性推进

1. `openclaw/openclaw` 官方 GitHub Release `v2026.4.10` 的 `published` 时间为 `2026-04-11T02:43:25Z`，折算北京时间为 `2026-04-11 10:43:25`，时间边界成立。  
2. 这版的新增非常工程化：加入 bundled Codex provider 和 plugin-owned app-server harness；新增可选 `Active Memory` 插件，在主回复前先跑 memory sub-agent；新增本地 `openclaw exec-policy` 命令，以及可让远端 gateway 客户端发现命令能力的 `commands.list` RPC。  
3. 同版还明显加码测试与安全：新增 Matrix / Telegram live QA lane，以及可在一次性 Linux VM 中运行 repo-backed QA 场景的 `--runner multipass`；同时收紧浏览器与 sandbox 导航防御、主机环境变量 denylist、插件安装依赖扫描、host media read 边界和多处 SSRF / 鉴权 / WebSocket 处理。

为什么值得工程团队关注：OpenClaw 这次不是单点 feature 发布，而是在把 agent 平台的“可记忆、可审批、可评测、可远端运维、可安全上线”一起补齐。对于自建 Agent 平台的团队，这比单独提升模型能力更接近真实上线成本。

最可能影响的环节：开发 / 测试 / 流程 / 安全治理 / Agent 平台。

【工程意义】
- 对开发：bundled Codex provider 与 Active Memory 可以减少自定义 provider 和记忆回填胶水，但也会引入新的运行时状态和调试面。
- 对测试：应补 Matrix / Telegram live QA、Multipass VM 回归、browser SSRF、防越权主机媒体读取、exec policy 同步和 gateway command discovery。
- 对流程：`exec-policy`、`commands.list` 和 live QA lane 更适合被纳入团队门禁，而不是只当个人本地功能。

【建议动作】
- 立即可做：检查内部 Agent 平台是否已经把 exec approval、host media read、plugin install dependency scan 和 browser SSRF 当成默认安全基线。
- 本周可验证：挑一条真实 agent workflow 跑一次 `--runner multipass` 或等价隔离环境回归，顺带验证命令发现、memory sub-agent 与远端 gateway 行为。
- 暂时观察：Active Memory 与 Codex provider 值得小流量试点，但不建议在未补齐可观测性前直接推到默认开启。

【来源】
- GitHub Releases：`openclaw/openclaw v2026.4.10`
